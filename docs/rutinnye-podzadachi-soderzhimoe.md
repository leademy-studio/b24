# Точное содержимое рутинных подзадач — Академия Спорта (GROUP_ID 29)

Каждая рутинная подзадача из месяца в месяц создаётся с **идентичным описанием** (проверено сравнением экземпляров за разные месяцы). Ниже — дословное содержимое (BBCode как в Bitrix24) каждого типа. Переменные части помечены как `{{…}}`; всё остальное — фиксированный boilerplate, переносимый без изменений.

> Замечание по разметке: часть задач хранится с HTML-тегами `<b></b>`, часть — с BBCode `[b][/b]`. Семантика идентична; ниже приведено как в исходных карточках.

>   - *Альтернатива* — собственный планировщик (уже есть недельный cron PPC): в `reminderAt` отправить исполнителю сообщение со ссылкой на задачу через `im.message.add` или `tasks.task.chat.message.send`. Минус — требует, чтобы генератор работал в нужный момент.

### 11. Передача конверсионных сигналов в площадки (feedback для автостратегий) — **еженедельная** (PPC)
_Новая стандартизированная задача (2026). Причина внедрения: работа уже выполняется, но **нерегулярно** — формализуется в рутину. В мире пакетных/автоматизированных РК (PMax, ЕПК с автотаргетом, tCPA/tROAS) качество и полнота конверсионных сигналов — главный рычаг человека: алгоритм учится на тех данных, которые ему передают. Текст-«рыба» полностью статичен (плейсхолдеров нет)._

**Расписание:** задача **создаётся каждый рабочий понедельник**, **дедлайн — старт +2 дня (среда)**. Если понедельник — нерабочий/праздничный день, старт сдвигается на ближайший рабочий день, дедлайн остаётся +2 дня от фактического старта. Каденция аналогична недельной задаче №10, но со сдвигом на начало недели (понедельник→среда vs среда→пятница), чтобы свежие конверсии попадали в площадки в начале недели. Генерируется отдельным недельным запуском под текущего месячного PPC-родителя.

```
[b]Задача:[/b]
Передать в рекламные площадки данные о качестве лидов / офлайн-конверсии из CRM за прошлую неделю, чтобы автостратегии обучались на реальной ценности заявок, а не на «сырых» лидах.

[b]Порядок выполнения:[/b]
Выгрузить из Bitrix24 CRM сделки/лиды за период с привязкой к рекламному источнику (yclid / gclid / ClientID, UTM)
Проставить по каждой записи статус качества (нецелевой / квалифицированный / продажа) и ценность (сумма сделки) для value-based стратегий
Сверить объём конверсий с Яндекс Метрикой / Google Analytics — зафиксировать расхождения, если есть
Загрузить офлайн-конверсии в площадки (см. спойлеры ниже)
Убедиться, что переданные цели/конверсии используются в стратегиях соответствующих кампаний (tCPA/tROAS)
Написать в комментарии: число переданных конверсий, период, обнаруженные расхождения

[b]Доступы и файлы:[/b]
В группе

[b]Загрузка офлайн-конверсий — Яндекс Директ[/b]
[spoiler]Через Яндекс Метрику: «Настройка» → «Загрузка данных» → «Офлайн-конверсии».
Сопоставление по ClientID или yclid; формат CSV с колонками: идентификатор, целевое действие (Target), дата-время (DateTime), стоимость (Price), валюта (Currency).
Для ценностного обучения (tROAS) обязательно заполнять Price реальной суммой сделки из CRM.
После загрузки проверить, что цель доступна в кампаниях и выбрана в стратегии.[/spoiler]

[b]Загрузка офлайн-конверсий — Google Ads[/b]
[spoiler]Offline Conversion Import по gclid (Tools → Conversions → Uploads) либо Enhanced Conversions for Leads (по хешированному email/телефону, если gclid не сохранился).
Колонки: Google Click ID, Conversion Name, Conversion Time, Conversion Value, Currency.
Убедиться, что соответствующее conversion action включено в столбец «Conversions» и используется стратегией (tCPA/tROAS).[/spoiler]

[b]ВАЖНО:[/b]
Задачу можно завершать только с комментарием-результатом, где указаны: число переданных конверсий, период и расхождения с веб-аналитикой (если были).
```

> Зависимость: задача опирается на корректно работающую сквозную аналитику (счётчики/цели/коллтрекинг — проверяются в недельной задаче №10) и на дисциплину заполнения источника в CRM-сделках. Если источник (yclid/gclid) не сохраняется — это блокер, выносится в отдельную тех. задачу на донастройку.

> **Условие генерации (параметр сделки).** Задача №11 имеет смысл только если клиент **может предоставлять данные** о качестве лидов/продажах (часть клиентов не отдаёт CRM-данные или работает в офлайне без оцифровки). Поэтому генерация гейтится полем сделки:
>
> | Параметр | Значение |
> |---|---|
> | REST-метод создания | `crm.deal.userfield.add` (scope `crm`) |
> | Код поля (FIELD_NAME) | `CONV_DATA` → `UF_CRM_CONV_DATA` (≤13 символов после префикса) |
> | Тип (USER_TYPE_ID) | `enumeration` — значения: **Да** / **Нет** / **Не выяснено** (`DEF` = «Не выяснено») |
> | Альтернатива | `boolean`-чекбокс «Клиент предоставляет данные о конверсиях» — проще, но не отличает «нельзя» от «ещё не спросили» |
>
> **Создано в CRM портала `leademy.bitrix24.ru`** (userfield ID `1407`). ID элементов списка для чтения значения в генераторе: `295` = Да, `297` = Нет, `299` = Не выяснено (значение по умолчанию). Генератор сравнивает `UF_CRM_CONV_DATA` сделки с `295`, чтобы решить, создавать ли №11.
>
> **Логика генератора:**
> - `Да` → спавнить №11 под месячного PPC-родителя.
> - `Нет` → №11 **не** создавать; отчёт PPC опирается только на платформенные конверсии (зафиксировать ограничение в карточке проекта).
> - `Не выяснено` (значение по умолчанию) → №11 не создавать, но создать **разовую задачу аккаунт-менеджеру** «выяснить у клиента возможность передачи офлайн-конверсий / качества лидов» — чтобы поле не висело пустым бессрочно.
>
> Рекомендация в пользу `enumeration` над простым чекбоксом: дефолт-чекбокс «снят» молча равен «нельзя» и навсегда отключает ценную задачу для клиентов, которых просто не успели опросить. Трёхзначный список делает «не выяснено» видимым и переводит его в действие.
