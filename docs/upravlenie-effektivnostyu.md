# Регламент: управление эффективностью и развитием сотрудников

Единый регламент по работе с командой агентства. Система-источник — Bitrix24 (`tasks.task.*`) + журнал гипотез; управленческая часть разбирается на планёрке.

**Состав:**
- §1–5 — **тест-задачи на вовлечённость** (мыслительный такт в начале месяца)
- §6 — **метрики эффективности** (что считаем из данных задач)
- §7 — **оценка задач (MARK)** (когда «+», когда «−»)
- §8 — **софт-скиллы** (поведение, отдельно от качества результата)
- §9 — **ownership-трек** (задачи на развитие самой компании)
- §10 — связь с автогенератором месяца

---

**Назначение (§1–5).** Ежемесячный «мыслительный такт» специалиста, который выполняется **до** того, как месяц заполнится рутинными подзадачами. Цель — выявлять истинные точки роста по каждому проекту и направлению, профилактировать «слепоту по проекту» из-за объёма рутины и развивать у специалиста мышление по своей области, а не только исполнение чек-листов.

**Контекст.** Рутинные подзадачи агентства максимально стандартизированы — это «рыбы»-константы с 0–4 плейсхолдерами (см. [rutinnye-podzadachi-soderzhimoe.md](rutinnye-podzadachi-soderzhimoe.md), [derevo-uslug-rutinnye-zadachi.md](derevo-uslug-rutinnye-zadachi.md)). Поэтому ~80% месяца — это исполнение шаблонов. Тест-задача — единственный обязательный такт, где специалист **думает над проектом сам и проактивно**, а не вытягивается на созвоне.

Формат описаний — BBCode Bitrix24 по стандарту [struktura-opisaniy-zadach.md](struktura-opisaniy-zadach.md). Связано с моделью генерации задач: [b24-routine-task-model] / [owner-dashboard-project].

---

## 1. Принципы

| Принцип | Почему |
|---|---|
| Создаётся **1–3 числа месяца**, до запуска рутины | Свежий взгляд, пока голова не забита чек-листами |
| Маленькая: **1–3 часа**, не нагрузка | Это диагностика мышления, а не дополнительный объём |
| Результат = **инсайт + гипотеза + метрика + действие + срок**, НЕ «сделано» | Проверяет понимание, а не исполнительность |
| Привязана к **1 проекту и 1 рычагу** | Заставляет приоритизировать, а не размазывать |
| Разбирается на ближайшей **планёрке** и **влияет на месячный план** | Вовлечённость растёт, только если вывод реально используется |
| Ответ кладётся в **журнал гипотез** | Замыкает петлю, позволяет отслеживать динамику |

**Дозировка.** 1–2 тест-задачи на специалиста в месяц, с ротацией из каталога (раздел 3). Больше нельзя — иначе сама тест-задача превратится в рутину.

---

## 2. Механика

1. **Где живёт задача.** Подзадача под месячным родителем проекта (`<Проект> — <Услуга> (<Месяц Год>)`), название с префиксом `[ТЕСТ] …`. Ставится тем же постановщиком, что и рутина.
2. **Куда кладётся результат.** Комментарий, помеченный как **результат** к задаче (по стандарту завершения, как у отчётов), + строка в **журнале гипотез** — едином файле учёта спорных страниц/гипотез по проектам (URL → описание → гипотеза → дата правки/переиндексации → метрика → ссылка на обоснование/ответ нейросети → динамика).
3. **Разбор.** На ближайшей планёрке 5–10 минут на специалиста. Вывод обязан попасть в месячный план работ (повлиять на состав рутинных подзадач или породить точечную задачу).
4. **Завершение.** Задача закрывается только с комментарием-результатом по структуре раздела 4 — по аналогии с правилом завершения отчётов и «Закупки ссылок».

---

## 3. Каталог тест-задач

Ротируются по месяцам. Ответственные по направлениям — по модели [b24-routine-task-model]: SEO/Администрирование → 101 (Максим Логвинов); PPC-стратегия → 17 (Святослав Павлишин); PPC-операционка → 103 (Владислав Павлишин).

### 🟦 SEO (отв. 101)

| № | Тест-задача | Что проверяет |
|---|---|---|
| S1 | **Точка роста месяца** — 1 проект, 1 страница/кластер с наибольшим потенциалом; обоснование данными (SerpStat / GSC / Вебмастер / Topvisor), гипотеза, целевая метрика и срок замера | Приоритизацию и работу с данными |
| S2 | **Чужими глазами** — экспресс-аудит проекта, который специалист обычно НЕ ведёт (или своего — как нового клиента); топ-3 проблемы | Лечит туннельное зрение |
| S3 | **Каннибализация / интент** — найти 1 пару каннибализирующих страниц или 1 несоответствие интента; предложить фикс (анкор / каноникал / перелинковка / целевая страница) | Понимание ранжирования, а не правок ради правок |
| S4 | **Дельта к конкуренту** — 1 конкурент из топ-10: что делает, чего нет у нас (структура, EEAT, кейсы, год/опыт), 1 действие к внедрению | Конкурентную насмотренность |

### 🟥 PPC (отв. 17 — стратегия; 103 — операционка)

| № | Тест-задача | Что проверяет |
|---|---|---|
| P1 | **«Будь это мой бюджет»** — назвать 1 самую неэффективную связку/площадку, предложить переаллокацию бюджета с прогнозом по CPL | Экономическое мышление |
| P2 | **Предвидь сезон** — спрогнозировать сдвиг спроса в нише на следующий месяц и предложить проактивный оффер/кампанию заранее | Проактивность, знание ниши клиента |
| P3 | **Одна структурная гипотеза** — вместо рутинной чистки предложить 1 тест (новый тип ЕПК, аудитория/ретаргетинг, оффер) с метрикой и сроком | Выход за пределы недельной оптимизации |

### 🟩 Сквозные (любое направление — диагностика вовлечённости)

| № | Тест-задача | Что проверяет |
|---|---|---|
| X1 | **3 вопроса, на которые я НЕ могу ответить про свой проект, но должен** | Способность увидеть собственные слепые пятна |
| X2 | **«Что дальше?»** — абзац стратегии по каждому проекту, как ответ клиенту на вопрос «что планируете и зачем» | Видение проекта целиком |
| X3 | **Обратный отчёт** — не «что сделал», а «что НЕ работает в проекте и почему» | Снимает оптимистичное искажение отчётности |

## 4. Структура результата (обязательная)

Любой ответ закрывается комментарием-результатом по единой рамке. Без всех пяти полей задача не считается выполненной.

```
[b]Проект / страница / кластер:[/b]
{{что именно разбирали — URL/кампания}}

[b]Что увидел (данные):[/b]
{{факт + источник: SerpStat / GSC / Вебмастер / Topvisor / Метрика / Директ}}

[b]Гипотеза:[/b]
{{почему так и что изменится, если сделать X}}

[b]Целевая метрика и срок:[/b]
{{какую метрику и на сколько ждём сдвинуть, дата контрольного замера}}

[b]Следующий шаг:[/b]
{{конкретное действие — правка / задача / тест}}

[b]Ссылка в журнале гипотез:[/b]
{{[url]…[/url]}}
```

## 5. Рубрика оценки вовлечённости

Применяется при разборе на планёрке. Динамика по месяцам даёт объективную картину вовлечённости и точек роста по каждому направлению.

| Уровень | Признак ответа |
|---|---|
| 🔴 Низкий | Пересказал рутину / «всё ок» / общие слова без данных |
| 🟡 Средний | Есть проблема и данные, но нет гипотезы, метрики или срока |
| 🟢 Высокий | Проблема + данные + гипотеза + метрика + срок + следующий шаг |

Отметка уровня фиксируется в журнале гипотез рядом со строкой задачи — так по месяцам видно, краснеет или зеленеет специалист по каждому проекту.

## 6. Метрики эффективности (для дашборда)

Метрики выполнения задач для отслеживания эффективности сотрудников. Считаются дашбордом из `tasks.task.list` (уровни 1–3) и из журнала гипотез / разбора (уровни 4–5). Технические запросы и поля — в [dashboard-tz.md](dashboard-tz.md) §10.

**Главный принцип — не мерить эффективность одним числом закрытых задач.** Это вознаграждает «перемолку рутины» и слепоту по проекту (закон Гудхарта). **Каждую метрику объёма обязательно парить с метрикой качества.**

### Уровень 1. Дисциплина исполнения (из сырых данных)

| Метрика | Формула | Источник |
|---|---|---|
| % в срок | закрыто до дедлайна / всего закрыто | `closedDate` vs `DEADLINE` |
| Средняя просрочка, дни | avg(`closedDate` − `DEADLINE`) по просроченным | те же поля |
| Активных просроченных сейчас | count(`counter.expired` > 0, статус ≠ 5) | `counter.expired` |
| Переносы дедлайна | число изменений `DEADLINE` за жизнь задачи | история задачи |

### Уровень 2. Производительность (объём — только в паре с качеством)

| Метрика | Формула | Источник |
|---|---|---|
| Закрыто/мес по типам | count по тегам: рутина / внеплановая / `[ТЕСТ]` | `TAGS`, `PARENT_ID`, `STATUS=5` |
| План/факт по нормативу | факт vs норматив пакета (SEO: N страниц, ТЗ, семантика) | модель [b24-routine-task-model] |
| WIP (в работе одновременно) | count(`STATUS=3`) на специалиста | `STATUS` |
| Доля внеплановых | внеплановые / всего | `TAGS` (реактивность проекта) |

### Уровень 3. Качество (это и есть эффективность, а не активность)

| Метрика | Формула | Источник |
|---|---|---|
| **% закрытий по стандарту результата** | `taskHasResult=Y` при `taskRequireResult=Y` | `WITH_RESULT_INFO` — самый дешёвый честный сигнал качества |
| Средняя оценка задачи | avg(`MARK`), доля «−» | `MARK` (см. §7) |
| % возвратов на доработку | возвраты в `STATUS=3` после сдачи / `STATUS=7` | `STATUS`/`subStatus`, история |
| Время на задачу vs норматив | `timeElapsed` vs ожидание по типу | `WITH_TIMER_INFO` |

### Уровень 4. Воздействие (результат, а не вывод задачи)

| Метрика | Источник |
|---|---|
| % подтвердившихся гипотез | журнал гипотез (тест-задачи) |
| Сдвиг проектных KPI | Topvisor (позиции), GSC/Метрика (трафик), Директ (CPL/заявки) |
| Удержание/продление подписки | CRM (регулярные сделки) |

### Уровень 5. Вовлечённость и софт-скиллы

| Метрика | Источник |
|---|---|
| Уровень тест-задачи 🔴/🟡/🟢 | рубрика §5 |
| Доля проактивных задач | инициированные специалистом, а не собственником |
| Предупреждённые переносы vs молчаливые | §8 |

### Как пользоваться

1. **Связки, а не одиночные числа.** «40 задач, 95% в срок, 90% с результатом, средний MARK +0.6» — портрет; «40 задач» — шум.
2. **Светофор по проекту, а не рейтинг людей.** Цель — видеть, где краснеет проект, а не наказывать. Иначе задачи закрывают пустыми (поэтому `taskHasResult` критичен — ловит этот гейминг).
3. **Уровни 1–3 — автоматически** из `tasks.task.list`; уровни 4–5 — полуручные (журнал гипотез + планёрка).
4. **Норматив по типам задач** проставляется один раз на каждый шаблон рутины (объём + время) — основа для «план/факт» и «время vs норматив».

## 7. Критерии оценки задач (MARK)

Поле `MARK` в Bitrix24 (`tasks.task.list` → `MARK`) — нативная оценка задачи постановщиком/аудитором при приёмке. Применяется и к рутине, и к тест-задачам по единой шкале. Основной принцип: **оценка ставится за качество результата относительно стандарта, а не за усилия и не за сам факт закрытия.** Критерии известны специалисту заранее.

### ✅ Положительная (+) — результат выше стандарта

| Критерий | Пример |
|---|---|
| В срок **и** с комментарием-результатом по стандарту | «Закупка ссылок» — финальные URL backlink'ов; отчёт прикреплён как результат; семантика — с 3 видами частотности |
| Принесла инсайт / точку роста сверх задания | при создании страницы заметил и зафиксировал каннибализацию или проблему интента, а не просто залил контент |
| Проактивно предотвратила проблему | предложил отыграть сезонный провал спроса **до** запроса клиента (урок «Рыжего слона» / фитнес май–июнь) |
| Тест-задача уровня 🟢 (см. §5) | гипотеза + данные + метрика + срок + следующий шаг |
| Результат измеримо сдвинул метрику | позиции / CPL / трафик выросли — подтверждено в журнале гипотез |
| Не потребовала доработок, готова к использованию | результат можно отдавать клиенту/собственнику без переписывания |

### ❌ Отрицательная (−) — результат ниже стандарта

| Критерий | Пример |
|---|---|
| Закрыта без обязательного комментария-результата (`taskHasResult=N` при `taskRequireResult=Y`) | «Закупка ссылок» без URL; отчёт без вложения; семантика без частотностей |
| Пришлось вернуть на доработку / переделать | возврат из сдачи обратно в статус «Выполняется» |
| Молчаливая просрочка без предупреждения | дедлайн сорван, заранее не предупредил |
| Исполнение «для галочки», без проверки | текст из нейросети без вычитки (кейс Натальи: разделительные линии, не проверено), пропущен пункт чек-листа |
| Фактические ошибки, дошедшие до клиента | неверные данные / правки на боевом сайте |
| Петля не замкнута | материал подготовил, но не передал / не уведомил — «по текстам нет обратной связи» |

### ⚪ Без оценки (нейтрально) — по умолчанию

Задача выполнена по стандарту, без выдающегося и без нарушений — **оценку не ставим**. Если хвалить за каждую рутину, «+» обесценивается и средняя оценка перестаёт быть сигналом. `MARK` остаётся индикатором исключений.

### Правила честности оценки

1. **Только зона ответственности специалиста.** Не ставим «−» за внешние причины: клиент не дал доступы, задержка бюджета, не согласован оффер.
2. **Объективность важнее строгости.** Критерий «−» — проверяемый факт (нет результата-комментария, был возврат, фактическая ошибка), а не «не понравилось».
3. **«−» всегда с комментарием-причиной.** Причина одновременно становится входом для разбора на планёрке / следующей тест-задачи.
4. **Систематика, а не разовость.** Один «−» — повод для фидбэка; устойчивый паттерн «−» по направлению — сигнал о точке роста или перегрузе, видимый на дашборде.

## 8. Оценка софт-скиллов

Софт-скиллы оцениваются **отдельным треком**, не через `MARK`. Принцип: `MARK` оценивает качество *результата* задачи, софт-скилл — *поведение* специалиста. Это разные оси, их нельзя смешивать.

### Базовое правило: своевременное предупреждение — это «+»

В агентстве есть проблема молчаливой коммуникации («по текстам нет обратной связи»). Если наказывать за честное «не успеваю, давай перенесём срок», команда обучается уходить в молчаливую просрочку. Поэтому **проактивная коммуникация поощряется, а не штрафуется**. Один и тот же факт «перенёс срок» оценивается по тому, *как* это произошло:

| Ситуация | Оценка |
|---|---|
| Предупредил **заранее** (до дедлайна) + причина + реалистичный новый срок, разово | ✅ плюс — ответственность и коммуникация |
| Предупредил **постфактум** / в день дедлайна, без причины | ⚪/− — поздняя коммуникация |
| **Молча просрочил**, перенос задним числом | ❌ минус (см. §7) |
| Просит переносить **почти каждую** задачу | 🔎 не моральный минус, а сигнал: перегруз / плохая оценка сроков / заниженные нормативы — разбирать причину |

### Правила трека

1. **Паттерн, а не поштучно.** Один перенос/инцидент ничего не значит — оценивается тренд за месяц/квартал.
2. **Не зашивать в `MARK`.** Поведение и качество результата — разные шкалы.
3. **Поощрять то, что хотим видеть.** Своевременное предупреждение о переносе = положительный сигнал; молчание = отрицательный.

### Измеримые прокси (для дашборда, из данных Bitrix)

| Прокси | Источник |
|---|---|
| Доля переносов, заявленных **заранее**, vs молчаливых просрочек | timestamp изменения `DEADLINE` vs старый дедлайн |
| Время первой реакции на задачу/комментарий | активность по задаче |
| Доля задач с просьбой переноса (нагрузка/планирование) | история изменений срока |

### Качественная рубрика (на планёрке, помесячно 🔴/🟡/🟢)

Не более 4–5 компетенций, иначе превращается в бюрократию:

- **Коммуникация** — предупреждает заранее, а не молчит
- **Работа с обратной связью** — принимает правки без сопротивления
- **Проактивность/инициатива** — сам приносит точки роста (связь с тест-задачами, §3)
- **Командность** — помогает коллегам, передаёт контекст
- **Надёжность планирования** — адекватно оценивает сроки

## 9. Ownership-трек: задачи на развитие компании

Отдельный трек для роста вовлечённости и **ответственности за развитие самой Leademy**. Механика общая с тест-задачами (§3) и софт-скиллами (§8): владелец + метрика + видимость + влияние на бизнес. Полигон — собственный сайт и продукты агентства: нет клиента, который накажет за эксперимент, а все точки роста, которые сейчас вытягиваются вручную на созвонах, специалист отрабатывает на своём проекте.

### Принцип: давать во владение исход, а не задачу

«Сделай страницу» — исполнение. «Ты отвечаешь, чтобы по нише *медицина* мы вышли в топ и получали лиды» — владение. Ответственность рождает второе: цель + свобода в способе + видимый результат. Задачи на развитие раздаются как **закреплённые зоны владения**, а не разовые поручения.

### Форматы задач

| Формат | Суть | Пример для Leademy |
|---|---|---|
| **Владение активом** | Закрепить за специалистом кусок самого агентства, один владелец | SEO собственного сайта; отраслевая ниша; стек инструментов (SerpStat MCP + GPT, журнал гипотез); база знаний |
| **Инициативный трек** | 1 предложение по улучшению компании в месяц + **право внедрить** | новый оффер, автоматизация рутины, новая услуга |
| **Эксперимент / R&D** | Гипотеза о росте агентства во владении, ведётся в журнале гипотез до метрики | «даёт ли отраслевой лендинг лиды дешевле контекста» (кейс со стоматологией) |
| **Кодификация экспертизы** | Превратить свой опыт в регламент/методичку для команды | стандарт по интенту / поведенческим / EEAT |
| **Участие в продукте** | Владение фичей внутреннего продукта | раздел метрик в дашборде, планировщик |
| **Кейс как продукт** | Довести клиента до результата и упаковать в публичный кейс | Академия Спорта, Yolo EKB |

### Механика (превращает задачу в ответственность)

| Условие | Почему обязательно |
|---|---|
| Один владелец, имя закреплено | Размытая ответственность = ничья |
| Автономия в способе | Владение невозможно без свободы решать |
| Метрика + срок | Ответственность измерима, иначе лозунг |
| Видимость на планёрке/дашборде | Признание подпитывает вовлечённость |
| Связь с результатом компании | Рост лидов/выручки → признание/бонус владельца |
| Право решать, а не только предлагать | Псевдо-владение демотивирует сильнее, чем его отсутствие |

### Чего избегать

1. **Владение ≠ бесплатная переработка.** Под зону развития выделяется время (часть месяца), иначе она проиграет клиентским дедлайнам и станет источником вины.
2. **Не раздавать всем всё.** 1 зона владения на человека — иначе размывается.
3. **Не наказывать за неудавшийся эксперимент.** R&D-владение работает, только если провал гипотезы — нормальный результат, а не «−» (см. §7).

## 10. Связь с автогенератором месяца (с июля 2026)

При создании месячного родителя проекта (Cloud Run `/api/cron/launch-month`, см. [owner-dashboard-project]) к набору рутинных подзадач добавляется **ровно одна** подзадача `[ТЕСТ] …` из каталога по ротации направления. Это единственная подзадача месяца, текст которой переменный (выбор из каталога), а не константная «рыба». В журнал гипотез строка может заводиться автоматически при создании задачи.

> Рекомендация для шаблона ежемесячного отчёта: добавить обязательный блок «Выводы и следующий рычаг» — тогда тест-задача и отчёт образуют единую петлю «гипотеза в начале месяца → проверка в отчёте».
