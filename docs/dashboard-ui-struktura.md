# UI-структура дашборда владельца

Информационная архитектура веб-интерфейса собственника. Опирается на модель данных и эндпоинты ТЗ [dashboard-tz.md](dashboard-tz.md) (§4 модель, §6 API, §8 планировщик, §9 Т‑Банк, §10 метрики) и регламент [upravlenie-effektivnostyu.md](upravlenie-effektivnostyu.md). Технически — SPA на Firebase Hosting за Firebase Auth (один пользователь-владелец, §2–3 ТЗ).

## Принципы интерфейса

- **Один взгляд = одно решение.** Главный экран отвечает на «всё ли в порядке и куда смотреть», детализация — по клику вглубь.
- **Сначала отклонения.** Просрочки, ошибки, частичные оплаты, красные метрики выводятся вверх; «всё ок» не шумит.
- **Действия рядом с данными.** «Развернуть месяц» и «внеплановая задача» — из карточки проекта, а не из отдельного меню.
- **Метрики людей — светофор по проектам, не публичный рейтинг** (см. §10 ТЗ, §6 регламента): акцент на динамике и точках роста, а не на ранжировании сотрудников.

## Навигация

Левое меню (или верхний бар), 7 разделов + сквозная лента событий:

```
Обзор · Проекты · Задачи · Сотрудники · Финансы · Подписки · Планировщик
                                                          [лента событий]
```

---

## 1. Обзор (главный экран)
**Источник:** `GET /api/overview?period=YYYY-MM`

- **KPI-плитки:** активных проектов · услуг в работе · % выполнения месяца · просрочек · сумма бюджетов · факт оплат.
- **Матрица «услуги × состояние»:** по SEO / PPC / Поддержке — сколько задач каждого типа активно / просрочено.
- **Алерты (вверху, красным):** просроченные задачи, ошибки cron-запусков, частичные/неразобранные оплаты.
- **Лента событий** (боковая колонка): оплаты, переходы сделок NEW→«В работе», ошибки планировщика, новые просрочки.

## 2. Проекты
**Источник:** `GET /api/projects` → `GET /api/projects/:groupId`

- **Список проектов:** карточки (название, активные услуги, % выполнения, индикатор просрочки, бюджет).
- **Карточка проекта:**
  - услуги месяца → дерево рутинных задач (статус, ответственный, дедлайн, флаги `overdue` / `ждёт контроля`);
  - прогресс-бар по подзадачам (`progress: done/total/percent`);
  - финансы проекта (бюджет из родителя месяца, сумма сделки, оплаты);
  - ссылка на журнал гипотез проекта.
- **Действия:**
  - `Развернуть месяц` → модалка → `POST /api/projects/:groupId/months` (`{service, package S/M/L, period, responsibleId, site, region, budget}`);
  - `Внеплановая задача` → модалка (шаблон №9) → `POST /api/projects/:groupId/ad-hoc-task`.

## 3. Задачи (сквозной операционный вид)
**Источник:** агрегация `tasks.task.list` (как в §5 ТЗ)

- Кросс-проектный список рутинных задач с фильтрами: тип / услуга / ответственный / статус / просрочка / `[ТЕСТ]`.
- Цель — «что горит» в целом по агентству, независимо от проекта.
- (опц.) inline-действия: комментарий / завершить (`POST /api/tasks/:id/comment`, `/complete`).

## 4. Сотрудники
**Источник:** `GET /api/staff-metrics?period=YYYY-MM&responsibleId=` (§10 ТЗ), журнал гипотез (уровни 4–5)

- **Нагрузка и распределение** (блок «заложить логику расчёта», 2026-06-21) — на каждого специалиста:
  - **Проектов на человека** — `count(distinct GROUP_ID)` по активным задачам (`STATUS<5`) из `tasks.task.list`.
  - **Задач в неделю** — среднее и фактическое (последняя неделя): закрытые задачи по `CLOSED_DATE`, группировка по ISO-неделям; активные WIP сюда не входят (их отражает «Проектов»).
  - **Время отписки по задаче** — циклическое среднее часа суток по `POST_DATE` из `task.commentitem.getlist` (фильтр `AUTHOR_ID`), корректно через полночь. **Второстепенный** фактор: позднее среднее (> норматива) повышает уровень внимания, но сам по себе не даёт «перегруз».
  - **Светофор нагрузки** — от нормативов (объём проектов + задач/нед, поправка на позднюю отписку); нормативы в проде — справочник Firestore (§10.3 ТЗ).
  - Реализация расчёта: `web/metrics.js` (`window.LeademyMetrics`) — чистые функции на форме ответов B24; роутер дёргает рендер по событию `screen:render`. Сейчас считает на встроенном демо-датасете, готов к привязке к `GET /api/staff-metrics`.
- **Карточка специалиста:** метрики **связками объём+качество** (закрыто/мес + % в срок + % закрытий по стандарту результата + средний `MARK` + % возвратов), тренд по месяцам, светофор по направлениям.
- **Тест-задачи и вовлечённость:** уровень 🔴/🟡/🟢 по месяцам (рубрика §5 регламента), доля проактивных задач, предупреждённые переносы vs молчаливые.
- **Ownership-зоны:** что закреплено за специалистом (§9 регламента) и статус.
- Без публичного ранжирования — динамика и точки роста.

## 5. Финансы
**Источник:** §4 ТЗ (бюджеты/сделки/товары) + §9 (Т‑Банк)

- Бюджеты и суммы сделок по проектам/услугам, агрегаты.
- **Лента платежей** (Т‑Банк): входящие, сопоставление со сделкой, переходы NEW→«В работе».
- **На ручной разбор:** частичные оплаты, нераспознанные платежи, несовпадение ИНН.

## 6. Подписки / Реестр
**Источник:** `GET /api/subscriptions` (read-only из регулярных сделок CRM)

- Представление «проект × услуги × пакет (S/M/L) × дата старта».
- Read-only: управление подписками/паузой остаётся в карточке сделки Bitrix.

## 7. Планировщик
**Источник:** логи `cron/launch-month` · `cron/launch-weekly` (§8 ТЗ)

- Статус и история запусков: создано / пропущено (идемпотентность) / ошибки, per-project.
- **Dry-run** (`{dryRun:true}`) — предпросмотр, что будет создано (важно для cutover 1 июля).
- Ручной перезапуск, индикатор отключения старых роботов Bitrix (чек-лист cutover §8.5).

## Решения по открытым вопросам (приняты 2026-06-17)

Данных из Bitrix/CRM достаточно для разделов 1–3, 5–7. Решения по остальному:

1. **Журнал гипотез** — встраивается в UI через **iframe** (Google Sheets). Раздел «Сотрудники» и блок тест-задач показывают журнал inline, отдельное хранилище для него не требуется.
2. **Хранилище ручных данных** — **Firestore**. Хранит то, чего нет в Bitrix: рубрику вовлечённости 🔴🟡🟢, ownership-владельцев, нормативы по типам задач (для «план/факт» и «время vs норматив»), а также дедуп `operationId` Т‑Банка и обработанные cron-запуски. ⚠️ Это **отклонение от изначального stateless-принципа** ТЗ (§1, §4): для HR/ручных данных вводится минимальное состояние. Нужно отразить в [dashboard-tz.md](dashboard-tz.md) (архитектура §3, модель §4).
3. **Внешние KPI проектов (уровень 4 метрик)** — **сторонние интеграции** (Topvisor / GSC / Метрика / Директ). Вне ядра; уровень 4 наполняется по мере подключения интеграций, до этого — ручной ввод/журнал.
4. **Канал уведомлений** — **личное сообщение в Bitrix24** в чат собственника. Технически: `im.message.add` (или `im.notify.personal.add`) с `DIALOG_ID` = user-id владельца. Требует добавить scope **`im`** к вебхуку (сейчас `crm, socialnetwork, task, tasks`). Триггеры: ошибка cron-запуска, новая/частичная оплата, рост просрочек.
5. **Визуальный язык** — **дизайн-код Leademy** (см. раздел ниже), тема **светлая**, шрифт **Google Sans Cyr**.
6. **Устройство** — **desktop-first** (как и сайт Leademy: компоненты для desktop, упрощение через `max-width`).

## Визуальный язык (дизайн-код Leademy)

Дашборд наследует дизайн-систему сайта Leademy — источник: `leademy-website/ldm-md/design-code.md`. Светлая тема, desktop-first. Ключевые токены:

| Категория | Токены / значения |
|---|---|
| Бренд-цвета | `--blue` `#14274e` (заголовки, кнопки, активное), `--blue-hover` `#1a356b` |
| Поверхности | `--white-smoke` `#f7f7f3`, `--numbers-bg` `#f5f6f9` (карточки/статистика), `--surface` `#fffdf9` |
| Текст | `--text-heading` (=blue), `--text-body` `#272727`, `--text-muted` `#999` |
| Sand-акцент | `--sand-100…500` — мягкие бордеры, hover, акценты |
| Радиусы | `--radius-sm 8px` (кнопки/compact), `--radius-md 12px` (inputs/таблицы), `--radius-lg 16px` (modal), `--radius-xl 20px` (карточки/крупные блоки) |
| Тени | `--shadow-sm` (карточка/dropdown), `--shadow-md` (hover), `--shadow-lg` (modal/overlay) |
| Типографика | `Google Sans Cyr`; шкала `--h1…h6`; body 14px/20px, `p` 1rem/1.6 |
| Контейнер | `1280px`, центрирование |

**Маппинг на компоненты дашборда:**
- **KPI-плитки / карточки проектов** — фон `--numbers-bg`, radius `--radius-xl`, `--shadow-sm`, hover `translateY(-4px)` + `--shadow-md` (паттерн `.sub-niches__item`).
- **Кнопки действий** — `.button` (синяя), `.button--light` (светлая), `.button--outline`; radius `--radius-sm`.
- **Таблицы** (метрики, задачи) — паттерн rich-text-table: header синий фон/белый текст, body 13–14px, hover-строка `rgba(26,53,107,.05)`, `overflow-x:auto`, radius `--radius-md`.
- **Модалки** (развернуть месяц, внеплановая задача) — паттерн `.lightbox-modal` + `.modal_small`: overlay `rgba(10,21,47,.62)` + blur, radius `--radius-lg`, поля `.field-input` (height 50px, radius 12px, focus-ring через `--blue-rgb`).
- **Светофор метрик/вовлечённости** — 🟢 успех / 🟡 внимание / 🔴 проблема: использовать семантические статус-цвета поверх sand-нейтралей, не пестрить.
- **Хедер дашборда** — паттерн `.navbar-container` (radius 8px, backdrop-blur, белый полупрозрачный фон).
- **Focus** — глобальный `:focus-visible` outline `rgba(var(--blue-rgb),.4)` (доступность).

**Правила фона (зафиксировано):**
- **Сайдбар (левая навигация)** — холодный нейтральный фон `#eceff4`; активный пункт — бренд-синий `--blue`.
- **Рабочая область** — белая (`#fff`).
- **Карточки** — `--numbers-bg #f5f6f9`; **панели** — белые с холодным бордером `#e3e7f0` / `--radius-lg`.
- **Хедер** — бренд-синий `--blue`, белый текст.
- **Песочные/бежевые оттенки (sand) — только тонкие акценты** (иконки, hover, разделители), **не заливки фона**. Фоновые поверхности — только холодные нейтрали и белый.
- **Светофор статусов** — семантические зелёный/янтарный/красный поверх нейтралей, не пестрить.

## Реализация (вёрстка) — начата 2026-06-21

**Источник дизайна:** Figma `wgYIO9vRthxdr2MhSzIxJg`, нода `1:2` (читается через Figma MCP `figma-developer-mcp`). Нода = **3 экрана** (`screen-overview`, `screen-project`, `screen-employees`) + тултипы Срок/Стандарт/MARK.

**Покрытие дизайном:** макеты есть для **3 из 7** разделов навигации. **Нет макетов:** Задачи, Финансы, Подписки, Планировщик, а также модалки «Развернуть месяц» и «Внеплановая задача» — свёрстаны по этому ТЗ на тех же компонентах (требуют дизайн-ревью).

**Где код:** статический SPA в `web/` (без сборщика/фреймворка — desktop-first, дизайн-токены Leademy).

| Файл | Назначение |
|---|---|
| `web/index.html` | каркас: сайдбар (логотип + 7 пунктов) + рабочая область |
| `web/styles.css` | дизайн-токены Leademy + компоненты (KPI, панели, таблицы, чипы, алерт, лента, дерево задач, прогресс, тулбар фильтров) |
| `web/app.js` | хэш-роутер: грузит `web/screens/<name>.html` партиалами в `<main>`; шлёт событие `screen:render` |
| `web/metrics.js` | логика расчёта нагрузки (проекты/чел, задачи/нед, время отписки) — `window.LeademyMetrics`, рендер в экран «Сотрудники» |
| `web/screens/overview.html` | §1 Обзор — **по Figma** |
| `web/screens/project.html` | §2 Карточка проекта — **по Figma** |
| `web/screens/employees.html` | §4 Сотрудники — **по Figma** |
| `web/screens/tasks.html` | §3 Задачи — по ТЗ |
| `web/screens/finances.html` | §5 Финансы — по ТЗ |
| `web/screens/subscriptions.html` | §6 Подписки — по ТЗ |
| `web/screens/scheduler.html` | §7 Планировщик — по ТЗ |

**Запуск превью:** `python3 -m http.server --directory web` (или конфиг `dashboard` в `.claude/launch.json`, порт 4174). Роутер использует `fetch` → нужен HTTP-сервер, на `file://` не работает.

**Как добавить страницу:** (1) создать `web/screens/<name>.html` на готовых классах (`.panel`, `.table`, `.kpi`, `.chip`, `.feed`, `.toolbar`); (2) пункт в `#nav` (`href="#<name>"`) в `index.html`; (3) `<name>` в массив `SCREENS` в `app.js`.

**Статус данных:** демо-данные зашиты в разметку, **без привязки к API** (`/api/overview` и т.д.). Следующий этап — подключение эндпоинтов §6 ТЗ.
