# План: автоматизация конспектов встреч и постановки задач

Улучшение операционной работы: автоматизировать цепочку **встреча → конспект → занесение в Google Docs → постановка задач команде в Bitrix24**. Документ фиксирует целевой процесс, архитектуру и этапы внедрения.

Источники форматов: [Princess Yachts — checklist-формат], [Yolo EKB — нумерованные списки], сырые расшифровки Яндекс Телемост (`docs/conspects/`). Связано с [dashboard-tz.md](dashboard-tz.md) (постановка задач в Bitrix) и моделью задач [b24-routine-task-model].

---

## 1. Текущий процесс (as-is) и проблемы

```
Яндекс Телемост Конспект → сырой .txt (спикеры + таймкоды + «вода»)
   → [вручную] промпт + .txt в нейросеть → текст-выжимка
   → [вручную] копирование в Google Docs «<Проект> — Конспекты встреч»
   → [вручную] создание таба (дата) + до-форматирование под визуальные требования
   → [вручную] постановка задач команде
```

**Проблема 1 — ручной труд.** Перенос текста в Docs и до-форматирование под требования (только нумерованные списки / checklist, без буллитов) съедают время на каждой встрече.

**Проблема 2 — непостоянный результат.** Один и тот же промпт + одна и та же расшифровка дают разный вывод. **Корневая причина:** нейросеть просят *одновременно извлекать смысл и форматировать* свободным текстом. Свободная генерация недетерминирована по форме.

---

## 2. Ключевое решение: разделить «извлечение» и «форматирование»

Не просить модель форматировать. Просить её **извлечь структурированные данные** (JSON по строгой схеме), а **форматирование делать кодом**. Тогда:
- **форма на 100% детерминирована** — её задаёт рендер-код, а не модель;
- модель отвечает только за содержание, и её вывод проверяем по схеме;
- один и тот же конспект всегда даёт одинаковое оформление.

```
сырой .txt
   │
   ▼  Этап 1 — ИЗВЛЕЧЕНИЕ (LLM, строгая JSON-схема)
структурированный JSON {meeting, themes[], tasks[]}
   │
   ▼  Этап 2 — РЕНДЕР (детерминированный код)
   ├──► Google Docs: новый датированный раздел + единое форматирование
   └──► Bitrix24: задачи команде из tasks[]
```

## 3. Архитектура пайплайна

Размещение — внутри того же сервиса, что и дашборд (Cloud Run, Node.js/TypeScript), переиспользуя `b24Call` и логику постановки задач из [dashboard-tz.md](dashboard-tz.md).

```
[Вход: .txt расшифровки]  (ручная загрузка в MVP; Telemost-автозабор — позже, §7)
        │
        ▼
[Cloud Run]  POST /api/conspect/ingest  { projectId, meetingType, participants[], date, rawText }
   1. ИЗВЛЕЧЕНИЕ → Claude (structured outputs, JSON-схема §4)
   2. ВАЛИДАЦИЯ JSON по схеме (ретрай при сбое)
   3. РЕНДЕР → Google Docs API (новый датированный раздел «ДД Месяц ГГГГ», §5)
   4. РЕНДЕР → черновики задач в Bitrix24 (§6) — с подтверждением владельца
   5. сохранить JSON в Firestore (источник правды, идемпотентность)
        │
        ▼
[Лента в дашборде]: «конспект создан, N задач предложено»
```

## 4. Этап 1 — извлечение в структурированный JSON (LLM)

### Модель и параметры
- **Модель:** `claude-opus-4-8` (Anthropic SDK `@anthropic-ai/sdk`). При желании удешевить высокочастотную обработку — выбор модели за владельцем (`claude-haiku-4-5`/`claude-sonnet-4-6`); по умолчанию Opus 4.8.
- **Structured outputs:** `output_config: { format: { type: "json_schema", schema: … } }` — гарантирует валидную форму ответа. (Параметр `temperature` на Opus 4.8 не используется — детерминизм обеспечивают схема + низкий effort + перенос форматирования в код.)
- **Effort:** `output_config: { effort: "low" }` (или `medium`) — извлечение по чёткой схеме не требует высокого.
- **Prompt caching:** системный промпт + 2 эталонных примера (Princess Yachts, Yolo EKB) ставятся в стабильный префикс с `cache_control` — стабилизирует стиль и удешевляет каждый прогон. Переменная часть (сырой .txt) — после последнего breakpoint.
- **Few-shot:** приложить оба конспекта-эталона как примеры «расшифровка → JSON» — это якорит, что считать «темой», «обязательством», «выполненной задачей».

### JSON-схема (черновик)
```json
{
  "type": "object",
  "additionalProperties": false,
  "properties": {
    "meeting": {
      "type": "object",
      "additionalProperties": false,
      "properties": {
        "project": { "type": "string" },
        "type": { "type": "string", "enum": ["client", "internal"] },
        "date": { "type": "string", "format": "date" },
        "participants": { "type": "array", "items": { "type": "string" } },
        "summary": { "type": "string" }
      },
      "required": ["project", "type", "date", "participants", "summary"]
    },
    "themes": {
      "type": "array",
      "items": {
        "type": "object",
        "additionalProperties": false,
        "properties": {
          "title": { "type": "string" },
          "project": { "type": "string" },
          "discussion": { "type": "string" },
          "tasks": {
            "type": "array",
            "items": {
              "type": "object",
              "additionalProperties": false,
              "properties": {
                "assignee": { "type": "string" },
                "direction": { "type": "string", "enum": ["","web_design","web_dev","seo","ppc","content","support","other"] },
                "text": { "type": "string" },
                "done": { "type": "boolean" },
                "clientCommitted": { "type": "boolean" },
                "deadlineProvisional": { "type": "boolean" },
                "deadline": {
                  "type": "object",
                  "additionalProperties": false,
                  "properties": {
                    "raw": { "type": "string" },
                    "kind": { "type": "string", "enum": ["none","absolute","relative_days","relative_weekday","recurring"] },
                    "weekday": { "type": "string", "enum": ["","mon","tue","wed","thu","fri","sat","sun"] },

  "required": ["meeting", "themes"]

```

### Промпт извлечения (переработка текущего)
Текущий промпт сохраняется по смыслу, но переориентируется на **извлечение, а не оформление**:
- убрать из промпта все правила форматирования (нумерация/checklist/запрет буллитов) — это теперь делает рендер;
- оставить правила обработки текста: игнорировать «воду», слова-паразиты, технические заминки, эмоции; вычленять суть/факты/решения; группировать в темы; внутри темы — выжимка + обязательства по участникам;
- **направление задачи:** проставлять `direction` (веб-дизайн / веб-разработка / seo / ppc / контент / поддержка) по сути задачи — для авто-маршрутизации ответственного (§6.2);
- **по срокам:** извлекать только *сигнал* (`deadline.kind/raw/...`), НЕ вычислять дату (см. §6); `deadlineProvisional=true`, если срок назван с оговоркой («ориентировочно», «примерно», «постараюсь»); `clientCommitted=true`, если срок озвучен клиенту как обещание;
- задача модели: «верни данные по схеме», а не «напиши документ».

> Ограничения по схеме structured outputs: не поддерживаются `minLength`/`maxLength`, числовые ограничения, рекурсия; для всех объектов нужен `additionalProperties: false`. SDK сам валидирует и подчищает несовместимые поля.

## 5. Этап 2 — рендер в Google Docs (детерминированный)

Форматирование задаётся **кодом**, поэтому одинаково всегда. Этот рендер — для `project` / `prospect` / `unassigned` (Google Docs). Для `internal` (внутренние планёрки) рендер другой — **BBCODE в профильный чат Bitrix24**, см. §6.4.

- **Авторизация:** сервис-аккаунт GCP с доступом к Google Docs/Drive API (Workspace domain-wide delegation либо явный шеринг файла на сервис-аккаунт). Секрет — в Secret Manager.
- **Файл проекта:** `«<Проект> — Конспекты встреч»` (Google Docs). Маппинг `projectId → documentId` — в Firestore. Для пресейла — отдельный документ «Пресейл — Конспекты встреч» (разделы по прослектам), см. §6.3.
- **Встреча = датированный раздел** (Heading 1 `ДД Месяц ГГГГ`) внутри одного документа проекта. См. решение по табам ниже.
- **Форматирование (единое, через Docs API `batchUpdate`):**
  - дата встречи — Heading 1; темы — Heading 2/3;
  - «Краткое резюме» — параграф;
  - **списки задач:** клиентские конспекты — **checklist** (Docs bullet preset `BULLET_CHECKBOX`), при `done: true` — отмеченный пункт; внутренние — **нумерованные** (`NUMBERED_DECIMAL`). Буллиты не используются — как в требованиях;
  - блок обязательств: подзаголовок `Задачи <Имя>:` + список задач этого участника.
- Выбор формата (checklist vs нумерованный) — по `meeting.type` или настройке проекта.
- **Порядок:** новый раздел вставляется сверху (после титула документа), чтобы последняя встреча была первой; либо в конец (проще). — мелкое UX-решение.

> **Решение по табам (проверено по докам Google, июнь 2026): создавать новые табы программно НЕЛЬЗЯ — никаким способом аутентификации.** Ни Google Docs REST API, ни Apps Script не имеют метода создания таба — оба только читают/пишут в **существующие** табы (REST — таргетинг по `tabId`; Apps Script — `getTab/getTabs/getActiveTab/setActiveTab`; методов `createTab`/`addTab` нет). **Сервисный аккаунт это не меняет** — он лишь способ аутентификации к тем же эндпоинтам, новых методов не добавляет (открытый feature-request Google от 09.2025 без ответа подтверждает отсутствие возможности). Поэтому модель «таб = встреча» не автоматизируема.
>
> **Выбранный подход — датированные разделы-заголовки в одном документе проекта.** Это единственный вариант, который (а) полностью автоматизируется одним `batchUpdate` (вставка текста + `updateParagraphStyle` + `createParagraphBullets`) и (б) сохраняет «один файл на проект с навигацией по датам» — заголовки встреч образуют структуру документа (outline-панель), выполняющую роль навигации вместо табов.
> Отвергнутые альтернативы: отдельный документ на встречу (теряет один файл на проект); ручное пред-создание пустого таба + запись в него по `tabId` (требует ручного шага — противоречит цели автоматизации).

## 6. Этап 3 — оценка дедлайнов и автопостановка задач в Bitrix24

### 6.1. Оценка дедлайнов (резолвер + движок реалистичности)

Принцип: **озвученный на созвоне срок не становится обязательством автоматически** — он проходит проверку данными и буфер. Цель — страховка от *оптимистичной оценки представителя агентства* (ошибка планирования), а не только от ошибок модели.

**Резолвер (код, между Этапами 1 и 3).** Берёт `deadline`-сигнал + **якорь = дата встречи** (из имени `.txt`/письма) → абсолютная дата:

| `kind` | Правило |
|---|---|
| `absolute` | взять дату; нет года → ближайший будущий относительно встречи |
| `relative_days` | дата встречи + N календарных дней |
| `relative_weekday` | ближайший `weekday` после встречи (`next` = +1 неделя) |
| `recurring` | **не дедлайн** — повторяющаяся/триггерная задача (ср. [b24-routine-task-model]) |
| `none` | дефолт-SLA = встреча + 3 рабочих дня, флаг «по умолчанию» |

Время **19:00 МSK**, перенос на рабочий день по производственному календарю РФ (`isdayoff.ru`) — общий хелпер с [dashboard-tz.md](dashboard-tz.md) §8.3. Неоднозначность («эта/следующая пятница») → флаг `ambiguous`, решает оператор.

**Движок реалистичности.** Для каждой задачи сверяет резолвенную дату с данными (источник — закрытые задачи Bitrix, тот же модуль метрик):
- **история по типу задачи** — p50/p80 реального цикла (`created→closed` / `timeElapsed`);
- **загрузка исполнителя** — открытые/просроченные задачи в окне;
- **ёмкость** — рабочих дней до срока vs суммарная оценка на исполнителе.

Вердикт **🟢 реалистично / 🟡 впритык / 🔴 нереалистично** с цифрами. Перегруз исполнителя в окне → 🔴 независимо от даты.

**Две даты + буфер.** `internalDeadline` (цель команды, по p50–p80 с учётом загрузки) и `clientDeadline` = internal + буфер (по исторической дисперсии, не best-case) + календарь. Клиенту озвучивается `clientDeadline`. Если устное обещание (`clientCommitted`) раньше — авто-🔴 и сигнал «переобещать».

**Где заслон:** всё это показывается в предпросмотре (§7) **до** создания задач — оператор видит обещание vs данные, правит и подтверждает. В Bitrix уходит `internalDeadline`; `clientDeadline` фиксируется отдельно. Петля рекалибровки буфера (обещано-vs-факт) — в [upravlenie-effektivnostyu.md](upravlenie-effektivnostyu.md) §6.

### 6.2. Постановка задач

Из `themes[].tasks[]` создаются задачи в группе проекта (переиспользуя постановку задач из [dashboard-tz.md](dashboard-tz.md) §6).

- **Маппинг исполнителя — по направлению задачи (`direction`), затем по имени.**
  Правило: направление задаёт ответственного по умолчанию; если в конспекте явно назван конкретный человек — он переопределяет (оператор подтверждает в предпросмотре). Таблица маршрутизации (`direction → RESPONSIBLE_ID`):

  | Направление | Ответственный |
  |---|---|
  | `web_design` (веб-дизайн) | **Денис Сафонов** (user **31**) |
  | `web_dev` (веб-разработка) | **Равиль Шакиров** (user **1**) |
  | `seo` | 101 Максим Логвинов |
  | `ppc` | 103 Владислав Павлишин (+ соисп. 17) |
  | прочее / не определено | по имени `assignee`; иначе — владелец/оператор |

  Справочник «имя → user_id» и «направление → user_id» — в Firestore. ID проверены на портале (2026-06-17): Денис Сафонов = 31, Равиль Шакиров = 1.
- **Проект → GROUP_ID:** из реестра проектов (регулярные сделки / настройка), как в дашборде.
- **Постановщик:** владелец; **заголовок:** `text`; **`DEADLINE` = `internalDeadline`** (после подтверждения оператором, §6.1); **описание:** ссылка на раздел конспекта + тема + `clientDeadline`, если был; по `done: true` задачу можно сразу закрывать или не создавать.
- **Подтверждение перед созданием:** в MVP — предпросмотр предложенных задач в дашборде, владелец подтверждает (а не слепое авто-создание). Позже — авто для доверенных типов.
- **Идемпотентность:** хеш (проект + дата + текст задачи) в Firestore — повторный прогон конспекта не плодит дубли.
- **Уведомление:** итог («создано N задач») — личным сообщением в Bitrix24 владельцу (`im.message.add`, как в [dashboard-tz.md](dashboard-tz.md) §8.6).

### 6.3. Потенциальные проекты (пресейл) — конспекты без проекта в CRM

Часть встреч — переговоры с **потенциальными** клиентами, у которых ещё нет проекта (workgroup) и регулярной сделки в «Производстве». Пайплайн это поддерживает явно.

**Тип привязки конспекта (`subjectType`)** — в записи ингеста (Firestore) и в списке «к разбору» каждый конспект классифицируется:
- `project` — существующий проект (workgroup из реестра) → как в §6.2;
- `prospect` — потенциальный клиент: **сделка в воронке продаж** (не «Производство»); если сделки ещё нет — создаётся из конспекта;
- `internal` — внутренняя встреча (без CRM-привязки);
- `unassigned` — **ни к чему не привязан** (классификацию отложили). Разбирается и рендерится в структурированный Google Doc как обычно, но отдельным документом, который кладётся в отдельную папку Google Drive (ниже); задач не порождает и ни за каким проектом/прослектом не закрепляется. Дальнейшую логику дорабатываем позже.

Запись ингеста несёт `subjectType` + `subjectId` (`GROUP_ID` | `leadId`/`dealId` | null).

**Непривязанные конспекты (`unassigned`) — отдельная папка Google Drive.** Этапы 1 и 2 выполняются как обычно: конспект **разбирается и рендерится в структурированный Google Doc** (то же форматирование, что в §5), но **отдельным документом на встречу**, а не разделом в проектном файле. Готовый Doc кладётся в выделенную Drive-папку «Конспекты без привязки». Пропускается только Этап 3 (задачи) и привязка к проекту/прослекту. Это сознательная «парковка»: переклассификация/привязка — на потом. В дашборде остаётся в списке со статусом «без привязки».

**Прослект = сделка в воронке продаж, а не отдельное хранилище.** Создаётся `crm.deal.add` в воронке продаж (категория ≠ 0). Оператор в списке «к разбору» выбирает существующую сделку **или** жмёт «Создать прослект» → заводится сделка из данных встречи (название/участники). Отдельный реестр прослектов не нужен.

**Задачи пресейла.** У прослекта нет GROUP_ID → задачи идут не под workgroup, а **с привязкой к сделке** через `UF_CRM_TASK` (формат `D_<dealId>`); исполнитель — по направлению (§6.2) или ответственный за продажи/владелец. Типичные: подготовить КП/смету, отправить материалы, демо, follow-up — те же `tasks[]`, оценка дедлайнов (§6.1) работает так же.

**Конспект в Docs.** Per-проектного документа ещё нет → отдельный документ **«Пресейл — Конспекты встреч»** с датированными разделами (заголовок = прослект + дата), та же модель разделов, что в §5.

**Конверсия прослект → проект (ключевой жизненный цикл).** Когда сделка выиграна и заводится проект (workgroup + регулярная сделка в «Производстве»), хранить маппинг `prospectDealId → projectGroupId` (Firestore). Тогда: новые конспекты по клиенту идут как `project`; прошлые пресейл-конспекты/задачи связываются с новым проектом (ссылка/перенос раздела) — история не теряется.

**Авто-привязка.** Матчинг по e-mail/домену участников: совпал проект → `project`; совпал лид/сделка продаж → `prospect`; ничего → предложить «новый прослект». Финальное решение — оператор.

> ✅ **Проверено на портале (2026-06-17):** есть выделенные **воронки сделок продаж** — `3 Квалификация`, `5 Первая продажа лицензии`, `7 Продажа продления`, `9 Продажа услуг` (помимо `0 Производство`). Лиды включены (~96), но это в основном авто-захваты с почты/Tilda — «сырой» верх воронки. **Решение:** прослект = **сделка в воронке продаж** (для агентских услуг — `9 Продажа услуг`, вход — `3 Квалификация`); лиды как источник прослекта не используем. **Конверсия прослект→проект** = перенос сделки в `0 Производство` + создание workgroup.

### 6.4. Внутренние планёрки → профильные чаты CRM (BBCODE; сначала задачи, потом конспект)

Отдельный поток для `internal`: конспект публикуется **не в Google Docs, а в профильный чат направления в Bitrix24** в формате BBCODE, с **нумерованными задачами и вшитыми ссылками на уже созданные задачи**, темы **сгруппированы по проектам** (`theme.project`).

**Ключевой порядок — обратный клиентскому: СНАЧАЛА задачи, ПОТОМ конспект.** Ссылки на задачи должны уже существовать, чтобы попасть в текст:
```
Внутренняя планёрка → .txt
  → Этап 1 (извлечение): задачи (direction/assignee/deadline) + группировка тем по проектам
  → СНАЧАЛА Этап 3: создать задачи в Bitrix (под workgroup проекта, ответственный по §6.2,
        дедлайны §6.1) → получить taskId + URL каждой
  → ПОТОМ рендер BBCODE: конспект с нумерованными задачами и [url]-ссылками на созданные задачи,
        темы по проектам
  → im.message.add(DIALOG_ID = <чат направления>, MESSAGE = BBCODE)
```

**Чат по направлению (`direction → DIALOG_ID`):**
| Направление | Чат |
|---|---|
| SEO | `chat1363` (Чат по SEO-продвижению) |
| PPC | `chat2045` (чат по PPC-продвижению) |
| прочие | ID чатов — уточнить |

Направление планёрки задаёт оператор (или извлечение). Если планёрка охватывает несколько направлений — блоки проектов соответствующего направления постятся в свой чат; MVP — один чат на планёрку, оператор выбирает.

**Формат BBCODE** (по текущему промпту внутренних планёрок, см. `docs/conspects/promts.txt`):
- названия тем (= проекты) — ЗАГЛАВНЫМИ + `[color=…]`; без форматирования по размерам;
- задачи — **только нумерованные** (`1. 2. 3.`), без буллитов;
- ссылка на задачу — `[url=<task_url>]…[/url]`; URL вида `https://leademy.bitrix24.ru/company/personal/user/<RESPONSIBLE_ID>/tasks/task/view/<taskId>/` (или групповой `/workgroups/group/<GROUP_ID>/tasks/task/view/<taskId>/`).

**Scope `im`** ✅ выдан (проверено 2026-06-17). Чаты подтверждены через `im.dialog.get`: `chat1363` = «SEO-продвижение», `chat2045` = «PPC-продвижение» — совпадают с направлениями. Постинг конспектов в чаты доступен. Задачи создаются как обычно — под workgroup проекта, ответственный по направлению (§6.2). Отличие от клиентского потока — **только** порядок (задачи раньше рендера) и канал публикации (чат вместо Docs).

## 7. Этап 4 (опц.) — автоматизация входа (сырой txt → БД)

**Ключевой факт (проверено по Telemost API v1.0.2 + докам Яндекс 360, июнь 2026):**
- **Telemost API НЕ отдаёт текст конспекта/расшифровки.** Спецификация (`api-specification.yaml`) покрывает только управление конференциями (`/conferences*`), со-организаторов и настройки организации (`/organizations/{org_id}/settings`). Эндпоинта для скачивания саммари/расшифровки/записи нет.
- **Конспект и текстовая расшифровка с таймкодами доставляются по email** организатору/участникам — это и есть наш сырой текст. **Запись** (видео) сохраняется на **Яндекс.Диск** организатора.
- Получатели саммари в настройках организации заданы **по ролям** (`summarization_email_receivers.value` ∈ `OWNER` / `INTERNAL_COHOST` / `INTERNAL_MEMBER`), **не произвольным e-mail** — поэтому ящик-приёмник должен принадлежать аккаунту-владельцу встреч (или со-организатору).

### Канал забора: переадресация письма на сервисный ящик (основной путь, MVP)

Самый простой автоматический путь — не давать сервису роль-получателя, а **переслать письмо Telemost фильтром-правилом** из почты владельца на отдельный сервисный ящик, который читает наш сервис.

```
[Telemost завершает встречу]
   → email (конспект + расшифровка) → ящик OWNER
        │  фильтр-правило: авто-пересылка писем от Telemost
        ▼
   → сервисный ящик (напр. Google Workspace/Gmail — под стек GCP)
        │  Gmail API watch + Pub/Sub (push) ИЛИ опрос (IMAP/poll)
        ▼
[Cloud Run] POST /api/conspect/ingest
   1. извлечь сырой текст: вложение .txt; если вложения нет — текст из тела письма
   2. сохранить в Firestore: rawText + дата + участники + тема + статус "к разбору"
   3. показать в дашборде в списке «Конспекты к разбору»
        │
        ▼
[Оператор] открывает конспект → выбирает привязку (проект / прослект / внутр. / без привязки) → запускает разбор
   → Этап 1 (извлечение) → предпросмотр конспекта + предложенных задач → подтверждение
   → Этап 2 (Docs) + Этап 3 (задачи в Bitrix / привязка к лиду-сделке для прослекта, §6.3)
```

Почему так проще:
- **обходит роль-ограничение** `summarization_email_receivers` — пересылка работает поверх любой настройки;
- **не требует Telemost API** для забора вообще;
- **список «к разбору» решает привязку** — оператор назначает проект/прослект/внутр. там же, где запускает разбор (email привязку не несёт; прослект можно создать на месте, §6.3);
- устойчивость к формату: берём вложение .txt, иначе — тело письма.

### Список «Конспекты к разбору» (UI)
Раздел в дашборде: входящие сырые конспекты со статусами `к разбору → разобран → задачи поставлены` (+ `без привязки`). По каждому: дата, участники, тема, кнопка «Разобрать» с **выбором привязки** — проект / **прослект** (выбрать или создать, §6.3) / внутренняя / **без привязки** (структурированный Doc отдельным файлом в Drive-папку, §6.3) → Этап 1. Это и приёмник, и точка ручной привязки, и предпросмотр перед записью в Docs/Bitrix.

**Предпросмотр задач (заслон по дедлайнам, §6.1):** каждая задача — строка `исполнитель · текст · internalDeadline (ред.) · clientDeadline (ред.) · 🟢/🟡/🔴 + цифры (история p80 / загрузка / ёмкость) · источник срока (из разговора / по умолчанию / ориентировочно)`. Оператор правит и подтверждает; 🔴/`ambiguous` требуют явного решения. Только после подтверждения — Этап 2 (Docs) и Этап 3 (Bitrix).

### Опционально (позже, для большей автоматизации)
- `POST /conferences` (Telemost API) — создавать встречи с **заголовком, кодирующим проект** (`<ProjectId> — тема`) → парсинг проекта из письма, авто-привязка без оператора.
- Матчинг по e-mail участников / домену клиента → проект (справочник в Firestore).
- `PUT /organizations/{org_id}/settings` — гарантировать роль-получателя саммари (если откажемся от пересылки).
- **Яндекс.Диск:** запись (видео) на Диске организатора — запасной источник, если понадобится аудио/видео; для txt канал — email.

### Формат письма Telemost (✅ подтверждено на реальном письме 12.06.2026)
- **Отправитель:** «Хранитель встреч»; **тема:** «Конспект встреч[и]» (проект в письме не указан → привязка через список «к разбору»).
- **Структура:** `multipart/mixed` → `multipart/related` (HTML-тело = собственное саммари Яндекса с встроенными картинками) + **вложение** `text/plain`, base64, `Content-Disposition: attachment`.
- **Вложение = сырая расшифровка** (то, что нужно Этапу 1). Имя файла: `ГГГГ-ММ-ДД ЧЧ:ММ (MSK) <meetingId>.txt`; содержимое начинается с `Встреча проходила ДД.ММ.ГГГГ с ЧЧ:ММ (MSK)…` — идентично файлам в `docs/conspects/`.
- **Ингест:** брать `.txt`-вложение (основное), тело письма — только страховочный фоллбэк; фильтр входящих по отправителю «Хранитель встреч»; `meetingId` из имени файла — ключ дедупа и (в перспективе) авто-привязки к проекту.

> Остаётся уточнить только доступ к сервисному ящику (Gmail API + Pub/Sub либо IMAP/app-password) и что авто-пересылка сохраняет вложение.

## 9. Этапы внедрения

1. **Извлечение (ядро):** эндпоинт `POST /api/conspect/ingest`, JSON-схема, промпт извлечения, structured outputs + few-shot + caching. Выход — валидный JSON. Проверить на 3 имеющихся расшифровках (Yolo 10.06, и др.).
2. **Рендер в Docs:** сервис-аккаунт + Docs API, единое форматирование (checklist/нумерация), решение по табам (§5). Идемпотентная запись.
3. **Дедлайны:** резолвер сигнала (дата встречи + календарь РФ) + дефолт-SLA. Базово — для постановки задач.
4. **Задачи в Bitrix:** маршрутизация по направлению (§6.2), предпросмотр и подтверждение, создание через batch, дедуп.
5. **Внутренние планёрки → чат:** обратный порядок (задачи → BBCODE-конспект со ссылками → `im.message.add` в чат направления, §6.4); нужен scope `im` и карта `direction→chatId`.
6. **UI:** список «Конспекты к разбору» (входящие) + предпросмотр конспекта и предлагаемых задач с дедлайнами + ручная загрузка .txt как фоллбэк; лента результата.
7. **Вход:** фильтр-правило авто-пересылки письма Telemost на сервисный ящик + ингест (Gmail API+Pub/Sub или IMAP); извлечение вложения .txt / тела письма. Сначала подтвердить формат письма.
8. **Движок реалистичности дедлайнов:** история p80 по типам / загрузка / ёмкость → 🟢🟡🔴, две даты + буфер (нужны накопленные данные закрытых задач). Зависит от модуля метрик.
9. **Рекалибровка буфера:** «обещано-vs-факт» по клиентским дедлайнам → авто-подстройка буфера (см. [upravlenie-effektivnostyu.md](upravlenie-effektivnostyu.md) §6).
10. **Авто-режим:** авто-привязка к проекту (заголовок встречи с ProjectId) и, для доверенных проектов, постановка задач без ручного подтверждения.
