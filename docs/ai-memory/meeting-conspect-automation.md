Воркстрим: автоматизировать цепочку **встреча → конспект → занесение в Google Docs → постановка задач команде в Bitrix24**. План: **docs/avtomatizatsiya-konspektov-plan.md**. Переиспользует инфраструктуру дашборда (Cloud Run + Firestore + b24-клиент) — см. [[owner-dashboard-project]].

Текущий ручной процесс: Яндекс Телемост Конспект → сырой .txt → шаблонный промпт → текст-выжимка → ручной перенос в Google Docs «<Проект> — Конспекты встреч» (табы по датам) → ручное до-форматирование → ручная постановка задач. Эталоны конспектов в репо: Princess Yachts (checklist-формат) и Yolo EKB (нумерованные списки); правило форматирования — только нумерованные/checklist, без буллитов.

**Ключевые находки/решения (проверены, июнь 2026):**
1. **Непостоянство результата** лечится разделением: LLM делает только **извлечение в строгий JSON** (Claude Opus 4.8, structured outputs `output_config.format`/json_schema, low effort, prompt caching системного промпта + 2 эталонов few-shot), а **форматирование делает код** (детерминированно). Промпт переориентировать на «верни данные по схеме», убрав правила оформления.
2. **Google Docs: создавать табы программно НЕЛЬЗЯ** — ни REST API, ни Apps Script, **ни через сервисный аккаунт** (это ограничение API-поверхности, не прав; есть только get/set существующих табов; открытый feature-request Google без ответа). Решение: **датированные разделы-заголовки (Heading 1 «ДД Месяц ГГГГ») в одном документе проекта** — единственный полностью автоматизируемый через `batchUpdate` вариант; checklist (клиент) / нумерация (внутр.) через bullet-presets `BULLET_CHECKBOX`/`NUMBERED_DECIMAL`.
3. **Telemost API (cloud-api.yandex.net/v1/telemost-api) НЕ отдаёт конспект/расшифровку** — только управление конференциями + настройки организации; саммари-получатели заданы по ролям (OWNER/INTERNAL_COHOST/INTERNAL_MEMBER), не произвольным e-mail.
4. **Сырая расшифровка приходит по EMAIL** (запись-видео → Яндекс.Диск организатора). **Формат письма подтверждён на реальном .eml (12.06.2026):** отправитель «Хранитель встреч», тема «Конспект встреч», `multipart/mixed`→`multipart/related` (HTML-тело = саммари Яндекса) + **вложение** `text/plain` base64 = сырая расшифровка, имя `ГГГГ-ММ-ДД ЧЧ:ММ (MSK) <meetingId>.txt`, начинается «Встреча проходила ДД.ММ.ГГГГ с ЧЧ:ММ (MSK)…».
5. **Этап 4 (забор txt) — выбранная схема:** фильтр-правило в почте владельца авто-**пересылает** письмо Telemost на сервисный ящик (Gmail/Workspace под GCP) → ингест берёт `.txt`-вложение → Firestore статус «к разбору» → список «Конспекты к разбору» в дашборде → оператор выбирает проект (email проект не несёт) → Этап 1. Telemost API для забора не нужен. Открытый вопрос — только доступ к ящику (Gmail API+Pub/Sub vs IMAP) и сохранность вложения при пересылке.

Маршрутизация задач из конспекта по направлению (`direction`): **веб-дизайн → Денис Сафонов (user 31)**, **веб-разработка → Равиль Шакиров (user 1)**; seo→101 Максим, ppc→103(+17). Направление задаёт ответственного по умолчанию, явно названный человек переопределяет. (ID проверены на портале 2026-06-17.) Конспекты могут быть `unassigned` (ни к чему не привязаны) → структурированный Google Doc отдельным файлом в Drive-папку «Конспекты без привязки», без задач.

Три потока вывода по `subjectType`: `project`/`prospect`/`unassigned` → Google Doc; **`internal` (внутренние планёрки) → BBCODE-пост в профильный чат направления Bitrix** (`im.message.add`, DIALOG_ID: SEO=chat1363 «SEO-продвижение», PPC=chat2045 «PPC-продвижение» — проверено 2026-06-17, scope `im` выдан, чаты доступны. Для internal **обратный порядок**: сначала создать задачи (получить taskId+URL), потом рендер конспекта с вшитыми `[url]`-ссылками; темы группируются по проектам, задачи нумерованы.

Прослект (потенциальный проект) = **сделка в воронке продаж**, не лид. Воронки сделок портала (crm.category.list entityTypeId=2, проверено 2026-06-17): 0 Производство, 3 Квалификация, 5 Первая продажа лицензии, 7 Продажа продления, 9 Продажа услуг. Для пресейла — `9 Продажа услуг` (вход — `3 Квалификация`); лиды (~96) = сырой верх воронки, как источник прослекта не используем. Задачи прослекта привязываются к сделке `UF_CRM_TASK=D_<dealId>`; конверсия = перенос сделки в 0 Производство + workgroup, маппинг prospectDealId→projectGroupId.

Оценка дедлайнов задач из конспекта (план §6.1): LLM извлекает только *сигнал срока* (kind/raw/...), дату вычисляет код-резолвер от якоря = дата встречи + календарь РФ (isdayoff, 19:00 МSK). Заслон от **оптимистичной оценки сроков представителем** (не клиента, а Равиля на созвоне): движок реалистичности (история p80 по типу задачи / загрузка / ёмкость → 🟢🟡🔴), две даты internalDeadline/clientDeadline + буфер, подтверждение оператором в предпросмотре; метрика «обещано-vs-факт» + авто-рекалибровка буфера — в docs/upravlenie-effektivnostyu.md §6 (Уровень 6).

**How to apply:** постановку задач из конспекта в Bitrix делать через тот же b24-клиент/логику, что и дашборд (см. docs/dashboard-tz.md §6; маппинг имя→RESPONSIBLE_ID, проект→GROUP_ID, дедуп). Связано с [[b24-routine-task-model]], [[staff-effectiveness-reglament]].
