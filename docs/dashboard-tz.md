# ТЗ: Дашборд владельца — мониторинг проектов и рутинных задач (GCP)

## 1. Контекст и цель

Владельцу бизнеса нужен **собственный веб-интерфейс** для контроля агентских проектов, чтобы не зависеть от UI Bitrix24. Дашборд показывает по каждому **проекту** и **услуге** (SEO / PPC) состав и состояние **активных рутинных задач** месяца, а также позволяет выполнять ключевые **действия**: развернуть месяц по услуге и создать внеплановую задачу по запросу с созвона.

**Дополнительно — с июля 2026 система берёт на себя запланированное создание рутинных задач** по активным проектам (раньше это делала автоматика внутри Bitrix). То есть сервис = дашборд (мониторинг + действия) **+ планировщик месячных запусков** (см. §8). Cutover — 1 июля 2026.

**Уровень независимости — 1** (см. [derevo-uslug-rutinnye-zadachi.md](derevo-uslug-rutinnye-zadachi.md) и обсуждение): свой UI и своя бизнес-логика поверх Bitrix; Bitrix остаётся системой-источником данных. Реестр активных проектов **уже есть в CRM** — регулярные сделки воронки «Производство» с полем услуг, пакетом (товар сделки) и связкой на GROUP_ID (§8.2). Собственного хранилища не требуется — система **stateless**, всё читается живьём из Bitrix/CRM REST.

Опорные документы: [derevo-uslug-rutinnye-zadachi.md](derevo-uslug-rutinnye-zadachi.md) (древо услуг), [rutinnye-podzadachi-soderzhimoe.md](rutinnye-podzadachi-soderzhimoe.md) (шаблоны №1–9), [struktura-opisaniy-zadach.md](struktura-opisaniy-zadach.md), [akademia-sporta-task-examples.md](akademia-sporta-task-examples.md) (структура месячного цикла), [dashboard-ui-struktura.md](dashboard-ui-struktura.md) (UI-структура интерфейса).

## 2. Пользователи и доступ

- Один пользователь — **владелец** (роль «admin/viewer» в одном лице). Возможность позже добавить менеджеров.
- Вход — **Firebase Auth (Google Sign-In)**, доступ ограничен whitelist'ом e-mail владельца. Бэкенд проверяет Firebase ID-token на каждом запросе.
- Действия (создание задач) доступны только аутентифицированному владельцу.

## 3. Архитектура (GCP)

```
[Браузер]
   │  Firebase ID-token (Google Sign-In)
   ▼
[Firebase Hosting]  ── статика UI (SPA: React/Vue/чистый JS) + rewrite /api/* → Cloud Run
   │  HTTPS, /api/*
   ▼
[Cloud Run: Node.js + Express]   ← основной бэкенд (расширение текущего server.js)
   ├─ middleware: проверка Firebase ID-token + whitelist
   ├─ b24 client (перенос b24Call + toFormPairs из server.js в src/bitrix24.ts)
   ├─ доменный слой: услуги, статусы, агрегации, шаблоны рутинных задач
   ├─ batch-вызовы Bitrix (создание подзадач пачкой, контроль rate-limit)
   └─ секрет вебхука ← Secret Manager (B24_WEBHOOK_BASE)
        │
        ▼
[Bitrix24 REST]  tasks.task.* · crm.product.* · socialnetwork.api.workgroup.list
```

Почему так: Cloud Run (а не «голая» Cloud Function) — потому что нужен полноценный сервис с роутингом, middleware авторизации и несколькими эндпоинтами; контейнер легко мигрирует с локального `server.js`. Firebase Hosting раздаёт статику и проксирует `/api/*` на Cloud Run (rewrite), что снимает CORS.

**Секреты:** `B24_WEBHOOK_BASE` (вебхук со scope `crm, socialnetwork, task`) — только в Secret Manager, монтируется в Cloud Run как переменная окружения при деплое. В репозитории/образе секрета нет.

## 4. Модель данных дашборда (вычисляется из Bitrix, не хранится)

```
Project (= workgroup / группа Bitrix)
  ├─ id (GROUP_ID), name, активность
  └─ services: ServiceMonth[]            // активные услуги текущего месяца

ServiceMonth (= глобальная месячная задача-родитель)
  ├─ taskId, service ∈ {SEO, PPC, TechSupport, Admin, Dev, Design}
  ├─ period (Месяц Год), responsibleId, deadline, status
  ├─ progress: { done, total, percent }  // по рутинным подзадачам
  └─ routineTasks: RoutineTask[]

RoutineTask (= подзадача-ребёнок)
  ├─ taskId, type ∈ {Закупка ссылок, Создание страниц, Семантика, ТЗ,
  │                  Перелинковка, Отчёт SEO, Запуск РК, Отчёт PPC, Внеплановая}
  ├─ title, status, responsibleId, deadline
  └─ flags: { overdue, awaitingControl }
```

**Финансы в мониторинге** (нужны):
- по проекту/услуге — **бюджет** из поля «Рекламный бюджет» родителя месяца и/или сумма сделки (`OPPORTUNITY`) и цены товаров (`crm.product.list` / `crm.deal.productrows`);
- агрегаты на overview: сумма бюджетов активных проектов, разрез по услугам, факт оплат (стадия сделки `WON`/входящий платёж — см. §9).

**Определения для мониторинга:**
- **Активная услуга месяца** = задача-родитель (`PARENT_ID` = 0/null) в группе со статусом ≠ 5 (Завершена).
- **Активная рутинная задача** = подзадача этого родителя со статусом ∈ {2 Ждёт, 3 Выполняется, 4 Ждёт контроля} (т.е. < 5).
- **Тип услуги** определяется парсингом названия родителя: `— SEO-продвижение`, `— PPC-сопровождение`, `— Тех.поддержка`, `— Администрирование`, `— Программирование`, `— Разработка дизайн-макета`.
- **Тип рутинной задачи** — по названию подзадачи (см. ID-списки в [rutinnye-podzadachi-soderzhimoe.md](rutinnye-podzadachi-soderzhimoe.md)): «Закупка ссылок», «Создание страниц», «Сбор семантического ядра», «Формирование ежемесячного отчета» и т.д.
- **Просрочка (overdue)** = `DEADLINE` < сегодня и статус < 5.

## 5. Маппинг на Bitrix REST

| Данные дашборда | Метод | Параметры |
|-----------------|-------|-----------|
| Список проектов | `socialnetwork.api.workgroup.list` | `filter:{">=ID":0}`, `select:[ID,NAME,TYPE,ACTIVE,CLOSED]` |
| Активные родители + подзадачи по проекту | `tasks.task.list` | `filter:{GROUP_ID, "<STATUS":5}`, `select:[ID,TITLE,STATUS,RESPONSIBLE_ID,DEADLINE,PARENT_ID,GROUP_ID,CLOSED_DATE]`, пагинация по `start/next` |
| **Реестр активных проектов** | `crm.deal.recurring.list` + `crm.deal.get` | базовые сделки cat 0; читать `UF_CRM_1727554217` (GROUP_ID), `UF_CRM_1725984512097` (услуги), `UF_CRM_1727619728` (сайт) |
| Каталог услуг (для действий) | `crm.product.list` | `select:[ID,NAME,ACTIVE,PRICE]` |
| Создание родителя/подзадач | `tasks.task.add` (через `batch`) | `fields:{TITLE,DESCRIPTION,RESPONSIBLE_ID,GROUP_ID,PARENT_ID,DEADLINE}` |
| (опц.) события статусов | вебхук `ONTASKUPDATE` → текущий [server.js](server.js) | push в кэш/лог |

Дерево «родитель→подзадачи» строим в бэкенде из плоского ответа `tasks.task.list` по `PARENT_ID` (как уже делалось при анализе группы 29).

## 6. API эндпоинты

**Чтение (мониторинг):**
- `GET /api/projects` — список проектов с краткой сводкой (сколько активных услуг, % выполнения, есть ли просрочки).
- `GET /api/projects/:groupId` — детально: услуги месяца → рутинные задачи со статусами/дедлайнами/ответственными.
- `GET /api/overview?period=YYYY-MM` — агрегат по всем проектам в разрезе услуг (сколько задач каждого типа активно/просрочено) — главный экран владельца.
- `GET /api/services` — древо услуг из CRM + маппинг на рутинные задачи.
- `GET /api/staff-metrics?period=YYYY-MM&responsibleId=` — метрики эффективности сотрудника/команды за период (см. §10): дисциплина, производительность, качество. Агрегируется из `tasks.task.list` с `params:{WITH_RESULT_INFO,WITH_TIMER_INFO}`.

**Действия (запись):**
- `POST /api/projects/:groupId/months` — развернуть месяц по услуге.
  Тело: `{ service: "SEO"|"PPC", package?: "S"|"M"|"L", period, responsibleId, site, region, budget }`.
  Логика: создать родителя + рутинные подзадачи по правилам пакета (SEO: N страниц = 3/5/10 → N×«Создание страниц», N×«ТЗ», по 1× семантика/перелинковка/закупка/отчёт; PPC: запуск РК + отчёт). Тексты — из шаблонов №1–8. Создание — через `batch`.
- `POST /api/projects/:groupId/ad-hoc-task` — внеплановая задача (шаблон №9, «обновление/запуск РК по запросу с созвона»).
  Тело: `{ parentTaskId?, title, what, platforms, period, budget?, materialsUrl?, withChecklist: bool }`.
- (опц.) `POST /api/tasks/:id/comment`, `POST /api/tasks/:id/complete` — если захотим закрывать/комментировать из дашборда.

**Планировщик и интеграции (служебные):**
- `POST /api/cron/launch-month` (см. §8) — вызывается Cloud Scheduler 1-го числа. Тело: `{ period?, dryRun?: bool }`. Перебирает активные регулярные сделки, для каждой идемпотентно создаёт месяц (та же логика, что `POST /months`). Авторизация — **только** по OIDC-токену сервис-аккаунта планировщика (не Firebase).
- `POST /api/cron/launch-weekly` (см. §8) — вызывается Cloud Scheduler каждую среду. Создаёт еженедельную задачу №10 «Оптимизация рекламы» под текущим PPC-родителем каждого активного PPC-проекта (дедлайн +2 дня/пт 19:00). Идемпотентность — по неделе+проекту.
- `POST /api/webhooks/tbank` (см. §9) — приёмник вебхука Т‑Банка о зачислении: парсит назначение, двигает сделку NEW→EXECUTING. Авторизация — по подписи/секрету Т‑Банка (не Firebase).
- `POST /api/cron/tbank-reconcile` — подстраховочный опрос выписки T‑Business (OIDC сервис-аккаунта).
- `GET /api/subscriptions` — представление реестра «проект × услуги × пакет» (read-only, агрегируется из регулярных сделок CRM; управление подписками/паузой — в карточке сделки Bitrix, а не у нас).

Пользовательские эндпоинты — за middleware Firebase-авторизации; `cron/*` — за проверкой OIDC сервис-аккаунта.

## 7. Шаблоны действий

Шаблоны рутинных задач хранятся в бэкенде как **константы-«рыбы»** (исходные тексты — в [rutinnye-podzadachi-soderzhimoe.md](rutinnye-podzadachi-soderzhimoe.md)) с плейсхолдерами:
- общие: `{{Месяц Год}}`, `{{регион}}`, `{{сайт}}`, `{{бюджет}}`;
- SEO «Создание страниц»/«ТЗ»: `{{N}}`, `{{ссылка на файл услуг}}`;
- PPC «Запуск/Обновление РК»: шапка (направление, площадки, офферы, период) + статичные чек-листы.

Правило количества для SEO задаётся пакетом (S=3 / M=5 / L=10). Внеплановая задача №9 **не входит** в авто-генерацию месяца — только через `ad-hoc-task` по кнопке.

## 8. Планировщик автозапуска рутинных задач (cutover: июль 2026)

С **июля 2026** внешняя логика становится ответственной за **запланированное создание** месячных задач по активным проектам (сейчас, в июне 2026, их ещё создаёт автоматика внутри Bitrix). Это переводит дашборд из «UI поверх Bitrix» в **систему-планировщик**.

### 8.1. Механизм

Две каденции генерации, оба триггера — Cloud Scheduler → Cloud Run (фон, OIDC сервис-аккаунта):

```
[Cloud Scheduler] МЕСЯЧНЫЙ  cron: 1-е число, ~00:05 Europe/Moscow
   ▼
[Cloud Run]  POST /api/cron/launch-month
   ├─ набор «активный проект × услуга» из регулярных сделок
   ├─ idempotent-создание родителей + месячных рутинных подзадач №1–8 (batch)
   ├─ перевод субподрядных сделок NEW→EXECUTING (§9.4)
   └─ отчёт (создано/пропущено/ошибки) → Cloud Logging + лента дашборда

[Cloud Scheduler] НЕДЕЛЬНЫЙ  cron: каждую среду, ~06:00 Europe/Moscow
   ▼
[Cloud Run]  POST /api/cron/launch-weekly
   ├─ для каждого активного PPC-проекта (услуга 141): найти текущего месячного
   │  PPC-родителя и создать подзадачу №10 «Оптимизация рекламы»
   │  (resp 103, соисполнитель 17), дедлайн = старт +2 дня, пятница 19:00
   └─ idempotent (одна задача оптимизации на проект в неделю), отчёт
```

### 8.2. Источник истины = регулярные сделки воронки «Производство»

Реестр активных проектов **уже существует в CRM** и его не нужно дублировать. Это **регулярные сделки** в воронке «Производство» (категория сделок **ID 0**, `isDefault=Y`). На 2026-06-10 их 12 (`crm.deal.recurring.list`), период — ежемесячно, `NEXT_EXECUTION=2026-07-01`.

Каждая сделка-проект несёт все нужные параметры в пользовательских полях:

| Поле сделки | Назначение | Пример (сделка 97, Phoenix) |
|-------------|-----------|------------------------------|
| `UF_CRM_1727554217` | **GROUP_ID проекта** в Bitrix (куда создавать задачи) | `"9"` |
| `UF_CRM_1725984512097` | **Перечень услуг** (множеств. список) | `[149,141,139]` |
| `UF_CRM_1727619728` | Сайт проекта | `["https://phoenix-yachts.com"]` |
| `UF_CRM_1726088408701` | Название проекта | `"PHOENIX YACHT MANAGEMENT"` |
| `UF_CRM_1722154664` | Дата старта сопровождения | `2023-12-01` |

**Справочник услуг** (значения `UF_CRM_1725984512097`) и привязка к наборам рутинных задач:

| ID | Услуга | Что создаём централизованно |
|----|--------|------------------------------|
| 139 | SEO-продвижение | родитель «<Проект> — SEO-продвижение (Месяц Год)» + рутинные подзадачи №1–6 (×N по пакету) |
| 141 | PPC-сопровождение | родитель «<Проект> — PPC-сопровождение (Месяц Год)» + подзадачи №7–8; внеплановые — №9 |
| 149 | Поддержка сайта | **только глобальный родитель** «<Проект> — Администрирование сайта (Месяц Год)»; авто-подзадач нет (заводятся вручную/по триггеру) |
| 147 / 143 / 145 / 151 / 287 / 293 | Аренда / SMM / Контент / Разработка / Комплекс / Пополнение балансов | **авто-задач не порождают** (биллинг/ручная постановка); вне скоупа автозапуска |

> Внимание на маппинг 149: услуга в сделке называется «Поддержка сайта», а глобальная задача — **«Администрирование сайта (Месяц Год)»** (как принято в проектах). Без рутинного набора подзадач — только родитель.

Алгоритм автозапуска: `crm.deal.recurring.list` (ACTIVE='Y', CATEGORY_ID=0) → для каждой базовой сделки прочитать поля выше + товарные строки → для каждой выбранной услуги создать родителя + его рутинные подзадачи в группе `GROUP_ID`.

#### Пакет SEO (S/M/L) — из товара сделки
Пакет определяется **товарной позицией сделки** (`crm.deal.productrows.get`) по `PRODUCT_ID`:

| PRODUCT_ID | Товар | Пакет | Страниц/мес (N) |
|-----------:|-------|:-----:|:---------------:|
| 87 | Оптимизация и продвижение web-сайта (Пакет S) | S | 3 |
| 1805 | … (Пакет M) | M | 5 |
| 1801 | … (Пакет L) | L | 10 |
| 499 / иное / нет строки | SEO без явного пакета | **S (дефолт)** | 3 |

N задаёт количество подзадач «Создание страниц» (×N) и «Создание ТЗ» (×N).

#### Ответственные — фиксированные правила (scope `user` подтверждён)
| Что | Ответственный (RESPONSIBLE_ID) | Соисполнитель (ACCOMPLICES) |
|-----|--------------------------------|------------------------------|
| SEO: родитель + **все** подзадачи | **101** — Максим Логвинов | — |
| PPC: **глобальная** задача (родитель) | **17** — Святослав Павлишин | — |
| PPC: **все** подзадачи | **103** — Владислав Павлишин | **17** — Святослав Павлишин |
| Администрирование сайта (услуга 149, только родитель) | **101** — Максим Логвинов | — |

При создании PPC-подзадач через `tasks.task.add`: `RESPONSIBLE_ID=103`, `ACCOMPLICES=[17]`.

> Итог: внешней логике **не нужно собственное хранилище** — пакет берётся из товара сделки, ответственные заданы правилами, остальное — из полей сделки. Система **stateless** (только runtime-кэш чтения, §9).

### 8.3. Дедлайн глобальных авто-задач

Дедлайн = **дата создания + 1 месяц + 3 рабочих дня** (даёт сотрудникам комфортный, но оперативный срок на отчёты).
- База: дата запуска (1-е число месяца) `+ 1 месяц` → 1-е число следующего месяца.
- Затем `+ 3 рабочих дня`, **пропуская выходные и государственные праздники РФ** (производственный календарь). Если стартовая дата (+1 мес) сама выпадает на нерабочий день — отсчёт 3 рабочих дней идёт от ближайшего рабочего.
- Время дедлайна — конец рабочего дня (напр. 19:00 МСК), часовой пояс портала Europe/Moscow.

**Источник календаря РФ:** внешний производственный календарь — API [isdayoff.ru](https://isdayoff.ru) (`GET /api/getdata?year=YYYY&...` → строка рабочих/выходных) с кэшированием результата на год; либо поддерживаемый в коде справочник праздников и переносов. Обязательно учитывать **переносы выходных** (постановления Правительства РФ на год) — поэтому статический список «суббота/воскресенье» недостаточен.

> Это изменение относительно текущего поведения: сейчас дедлайн = ровно 1-е число след. месяца (см. задачу 5743). Новое правило сдвигает его на +3 рабочих дня.

**Дедлайн еженедельной задачи «Оптимизация рекламы» (№10):** проще — **старт (среда) + 2 дня, 19:00** (пятница), по стандартной логике Bitrix24 «+N дней». Производственный календарь РФ здесь **не** применяется (фиксированный сдвиг +2 дня).

### 8.4. Идемпотентность (защита от дублей)
Перед созданием родителя — проверка `tasks.task.list` по `GROUP_ID` + точному `TITLE` `«<Проект> — <Услуга> (<Месяц Год>)»`. Если уже существует — **пропустить** (и залогировать). Аналогично для подзадач. Запуск должен быть безопасен при повторном вызове (ретрай Scheduler, ручной перезапуск).

### 8.5. План перехода (cutover на 1 июля 2026)

Текущая автоматика Bitrix состоит из **двух слоёв роботов**, которые внешняя логика заменяет:
- **Слой A** — робот на сделке воронки «Производство»: при наличии выбранных услуг создаёт глобальную задачу по направлению с дедлайном +1 мес.
- **Слой B** — роботы внутри задач, порождающие рутинные подзадачи (их сейчас приходится **копировать вручную** по группам, и они **зацикливаются** в списке запущенных БП).

Регулярные **сделки** (биллинг/реестр) **остаются** — отключаем только роботы-генераторы задач. **Роботы отключаются вручную аккаунтом владельца** (подтверждено).

1. **Инвентаризация роботов** в воронке «Производство» (cat 0): какие роботы на стадиях создают задачи (Слой A), и какие роботы/шаблоны в задачах плодят подзадачи (Слой B). Зафиксировать.
2. **Dry-run внешнего генератора на июль:** `launch-month {dryRun:true}` читает 12 регулярных сделок и выводит, что было бы создано (проект → услуги → родители + подзадачи), без записи. Сверить с тем, что делает текущая автоматика.
3. **Отключить роботов Слоя A и B** (приостановить, не удалять — для отката) — строго **до 1 июля**, иначе дубли. Это же убирает зацикливание старых шаблонов.
4. **Включить Cloud Scheduler** с 1 июля. Первый боевой запуск централизованно создаёт июль 2026 по всем активным сделкам.
5. **Сверка**: число родителей/подзадач = ожидаемому из сделок; per-project отчёт, алерт при расхождении.
6. Июнь 2026 — последний месяц «по-старому» (уже в Bitrix).

> **Критический риск:** если включить внешний планировщик, не отключив роботов Слоя A/B, на 1 июля каждый проект получит **два** комплекта задач. Шаги 3→4 строго последовательны; идемпотентность (§8.4) — вторая линия защиты, но не замена отключения роботов.

### 8.6. Надёжность планировщика
- Запуск НЕ зависит от того, открыт ли дашборд (Cloud Scheduler → Cloud Run работает в фоне).
- Ошибки части проектов не должны валить весь запуск: обработка по проектам независимая, агрегированный отчёт с per-project статусом.
- Ретрай Scheduler + идемпотентность; алерт владельцу при ошибках (e-mail/чат).
- Часовой пояс портала — Europe/Moscow (дедлайны и `00:03` создания в MSK); cron настроить в нём же.

## 9. Авто-перевод сделки NEW → «В работе» (платёж Т‑Банк / субподряд)

Автоматический перевод сделки со стадии **«Получение оплаты от клиента» (NEW) → «В работе» (EXECUTING)** по двум независимым основаниям:
- **по входящему платежу** клиента (§9.1–9.3) — для обычных проектов;
- **по признаку «Субподряд»** (§9.4) — для субподрядных проектов, без ожидания платежа.

### 9.1. Триггер и сопоставление

Главный ключ сопоставления — **номер счёта из назначения платежа** (счёт генерируется из сделки). Пример назначения:
> «Оплата по счету **№115-1119-06-2026** от 01.06.2026 за настройку и ведение контекстной рекламы НДС не облагается»

**Формат номера счёта (подтверждён): `{CLIENT_ID}-{SELF_ID}-{MONTH}-{YEAR}`**, где `CLIENT_ID` = **COMPANY_ID** клиента, `SELF_ID` = **DEAL_ID**. Проверено на примере: `115` = company 115 (ООО «РЕЛАКС»), `1119` = deal 1119 («КР-18», cat 0, стадия NEW), её COMPANY_ID = 115. Номер даёт сразу и сделку, и компанию.

```
[Т‑Банк (Т‑Бизнес) API]  входящий платёж: { ИНН плательщика, сумма, дата, назначение }
        │  (вебхук о зачислении ИЛИ периодический опрос выписки)
        ▼
[Cloud Run] POST /api/cron/tbank-payments   (или обработчик вебхука)
   1. Распарсить номер из назначения: regex №\s*(\d+)-(\d+)-(\d{2})-(\d{4})
      → CLIENT_ID=115 (COMPANY_ID), SELF_ID=1119 (DEAL_ID), 06, 2026
   2. crm.deal.get(SELF_ID) → проверить: CATEGORY_ID==0, STAGE_ID=="NEW",
      COMPANY_ID == CLIENT_ID
   3. Доп. верификация (анти-фрод): ИНН плательщика == RQ_INN компании CLIENT_ID
      (crm.requisite.list {ENTITY_TYPE_ID:4, ENTITY_ID:CLIENT_ID})
   4. Проверка суммы: сумма платежа >= сумма сделки (OPPORTUNITY).
      Если меньше → ЧАСТИЧНАЯ оплата → НЕ двигать, залогировать, оставить в NEW.
   5. crm.deal.update {id:SELF_ID, STAGE_ID:"EXECUTING"}        // NEW → «В работе»
   6. лог + запись в ленту дашборда (кто, сколько, какая сделка, какой счёт)
```

**Фоллбэк, если номер не распарсился:** INN плательщика → `crm.requisite.list {ENTITY_TYPE_ID:4, RQ_INN}` → COMPANY_ID → `crm.deal.list {CATEGORY_ID:0, COMPANY_ID, STAGE_ID:"NEW"}`; двигать только при **единственном** кандидате, иначе — на ручной разбор. Свой ИНН агентства (7203566588 / ООО «ЛИДЕМИ») исключать.

Подтверждённые в портале значения:
- Воронка «Производство» = категория **0**; стадии `NEW`="Получение оплаты от клиента", `EXECUTING`="В работе".
- ИНН — в реквизитах компании: `crm.requisite`, `RQ_INN`, `ENTITY_ID`=COMPANY_ID.
- Счета = смарт-процесс «Счёт» (`entityTypeId 31`), привязка к сделке через `parentId2`=DEAL_ID; `accountNumber` — порядковый, не равен печатному номеру.

> Печатный номер не хранится в полях счёта (только в печатной форме) — поэтому матчинг идёт парсингом назначения. Но т.к. оба ID (компания и сделка) зашиты в номер, а шаги 2–3 их перекрёстно проверяют, ложное срабатывание исключено. Риск — смена шаблона нумерации счёта: завязать тест на стабильность формата.

### 9.2. Источник событий Т‑Банк (T‑Business OpenAPI)
- **API:** T‑Business OpenAPI, база `https://business.tbank.ru/openapi/api/`, авторизация `Authorization: Bearer <TOKEN>`, `Accept: application/json`. Токен — в **Secret Manager** (отдельно от вебхука Bitrix).
- **Способ получения событий — вебхук о зачислении** (требование владельца): подписаться в T‑Business OpenAPI на событие входящей операции; T‑Банк шлёт POST на наш `POST /api/webhooks/tbank` при поступлении средств.
  - Эндпоинт вебхука **публичный** → обязательна проверка подлинности (подпись/секрет T‑Банка), иначе любой сможет двигать сделки.
  - Из payload берём: ИНН плательщика, сумму, дату, **назначение платежа** (для парсинга номера счёта §9.1).
- **Подстраховка:** периодический опрос выписки (operations/statement) раз в N минут через Cloud Scheduler для платежей, по которым вебхук не дошёл; дедуп по `operationId`.
- Учитывать только **входящие/кредитовые** операции; отфильтровывать возвраты, комиссии, внутренние переводы.

### 9.3. Граничные случаи
- **Частичная оплата:** сумма платежа < суммы сделки (OPPORTUNITY) → **не двигать** (оставить в NEW), залогировать. Перевод только при полной оплате одним платежом.
- **Несовпадение ИНН и сделки из номера:** ИНН плательщика ≠ ИНН компании сделки → не двигать, на ручной разбор.
- **Идемпотентность:** один платёж не двигает сделку дважды — хранить обработанные `operationId` (минимальное состояние); вебхук может прийти повторно.
- Платёж от **неизвестного ИНН** / номер не распарсился / сделка не в стадии NEW → в лог на ручной разбор, без действия.

> Безопасность: ключ Т‑Банка — доступ к банковским данным; хранить только в Secret Manager, эндпоинт вебхука защитить подписью/секретом от Т‑Банка, опрос — по сервис-аккаунту.

### 9.4. Субподрядные сделки — авто-переход без оплаты

Часть проектов агентство ведёт **на субподряде** — по ним нет прямого входящего платежа от клиента с совпадающим ИНН (оплата приходит от подрядчика-заказчика иначе), поэтому матчинг §9.1 для них не сработает. Такие сделки должны переходить **NEW → «В работе» автоматически**, без ожидания платежа.

**Поле сделки (создано):** чекбокс **«Субподряд»** — `UF_CRM_SUBCONTRACT` (тип `boolean`, ID 1403, `XML_ID=SUBCONTRACT`, по умолчанию «Нет»). Значение `1`/`Y` = проект на субподряде.

**Правило перехода:** сделка в категории 0, стадия `NEW`, `UF_CRM_SUBCONTRACT` = Y → автоматически `crm.deal.update STAGE_ID="EXECUTING"`, независимо от платежей. **Срабатывает только на запуске планировщика 1-го числа** (§8): при обработке регулярных сделок, если сделка субподрядная и в NEW — сразу двигаем в EXECUTING (периодический проход не требуется).

**Взаимодействие с §9.1:** обработчик платежей **пропускает** субподрядные сделки (их перевод не завязан на платёж). Признак субподряда читается из `UF_CRM_SUBCONTRACT` при `crm.deal.get`.

## 10. Метрики эффективности сотрудников (аналитический модуль)

Аналитический слой поверх мониторинга: метрики выполнения задач для отслеживания эффективности сотрудников. Управленческая трактовка и критерии — в регламенте [upravlenie-effektivnostyu.md](upravlenie-effektivnostyu.md) §6–9; здесь — техническая реализация.

**Принцип (закладывается в UI):** эффективность не сводится к числу закрытых задач — каждую метрику объёма показывать в паре с метрикой качества (защита от гейминга «закрыл пустую задачу»). Дашборд показывает **светофор по проекту/направлению**, а не публичный рейтинг людей.

### 10.1. Источник данных

Один запрос на сотрудника/период:
```
tasks.task.list
  filter: { RESPONSIBLE_ID, ">=CLOSED_DATE": <начало периода>, "<=CLOSED_DATE": <конец> }
          (+ отдельный проход по активным: "<STATUS":5 для WIP/просрочек)
  select: [ID, TITLE, STATUS, subStatus, RESPONSIBLE_ID, ACCOMPLICES,
           CREATED_DATE, DEADLINE, CLOSED_DATE, MARK, TAGS, GROUP_ID, PARENT_ID, COUNTERS]
  params: { WITH_RESULT_INFO: true, WITH_TIMER_INFO: true }
```
Поля-ключи: `taskRequireResult`/`taskHasResult` (закрытие по стандарту результата), `MARK` (оценка ±), `counter.expired` (просрочка), `timeElapsed` (затраченное время), `closedDate` vs `DEADLINE` (срочность). Переносы дедлайна и возвраты в работу — из истории задачи (при необходимости — отдельный источник/лог `ONTASKUPDATE`).

### 10.2. Метрики по уровням

| Ур. | Метрика | Формула / поле |
|----|---------|----------------|
| 1 Дисциплина | % в срок | `closedDate ≤ DEADLINE` / всего закрытых |
| 1 | средняя просрочка, дни | avg(`closedDate − DEADLINE`) по просроченным |
| 1 | активных просроченных | count(`counter.expired>0` & `STATUS<5`) |
| 1 | переносы дедлайна | число изменений `DEADLINE` (история) |
| 2 Производительность | закрыто/мес по типам | count(`STATUS=5`) с группировкой по `TAGS`/`PARENT_ID` |
| 2 | план/факт по нормативу | факт vs норматив типа задачи (§7 шаблоны) |
| 2 | WIP | count(`STATUS=3`) |
| 2 | доля внеплановых | внеплановые / всего (`TAGS`) |
| 3 Качество | **% закрытий по стандарту** | `taskHasResult=Y` / `taskRequireResult=Y` |
| 3 | средняя оценка / доля «−» | avg(`MARK`), count(`MARK<0`) |
| 3 | % возвратов на доработку | переходы в `STATUS=3` после сдачи / `STATUS=7` |
| 3 | время vs норматив | `timeElapsed` vs ожидание по типу |
| 4 Воздействие | % подтвердившихся гипотез | журнал гипотез (вне Bitrix) |
| 4 | сдвиг KPI проекта | Topvisor / GSC / Метрика / Директ (внешние) |
| 5 Вовлечённость | уровень тест-задачи 🔴🟡🟢, проактивность, переносы | журнал гипотез + история (см. §8–9 регламента) |

Уровни 1–3 считаются автоматически из Bitrix; 4–5 — полуручные (журнал гипотез/планёрка), на дашборде показываются как вводимые/импортируемые значения.

### 10.3. Нормативы по типам задач

Для метрик «план/факт» и «время vs норматив» нужен справочник нормативов на каждый шаблон рутины (ожидаемые объём и время) — задаётся один раз в конфиге бэкенда рядом с шаблонами-«рыбами» (§7), источник типов — [rutinnye-podzadachi-soderzhimoe.md](rutinnye-podzadachi-soderzhimoe.md).

### 10.4. UI

Экран «Сотрудники/Команда»: на каждого специалиста — карточка со связками метрик (объём + качество рядом), тренд по месяцам, светофор по направлениям. Без публичного ранжирования людей — акцент на динамике и точках роста по проектам.

## 11. Нефункциональные требования

- **Rate-limit Bitrix** (~2 rps, `QUERY_LIMIT_EXCEEDED`): чтение — параллельно с ограничением конкурентности; запись — через `batch` (до 50 команд/запрос) + последовательная очередь. Реализовать ретрай с backoff на `QUERY_LIMIT_EXCEEDED`/`OVERLOAD_LIMIT`.
- **Кэш чтения**: in-memory кэш ответов `tasks.task.list` на 30–60 сек (Cloud Run instance) — чтобы дашборд не бил Bitrix на каждый рефреш. Опционально — обновление по вебхуку `ONTASKUPDATE`.
- **Cold start / stateless**: никакого состояния в памяти кроме кэша; конфиг и шаблоны — в коде/Secret Manager.
- **Логирование**: Cloud Logging; маскировать код вебхука (как `maskWebhookBase` в [server.js](server.js)).
- **Безопасность**: вебхук — ключ от всего портала; доступ к дашборду строго по whitelist, ID-token проверяется на каждом запросе.

## 12. Что переиспользуем из репозитория

## 13. Этапы реализации

1. **Рефакторинг клиента**: вынести `b24Call`/`toFormPairs` в `src/bitrix24.ts`; покрыть тестом на пагинацию.
2. **Read-API**: `GET /projects`, `/projects/:id`, `/overview` + сборка дерева и агрегаций; in-memory кэш.
3. **UI-мониторинг**: SPA — экран overview (услуги×проекты, прогресс, просрочки) и карточка проекта.
4. **Авторизация**: Firebase Auth + middleware проверки токена и whitelist.
5. **Action-API**: `POST /months` (генератор по шаблонам и пакету, batch, идемпотентный) и `POST /ad-hoc-task`.
6. **Планировщик**: эндпоинты `cron/launch-month` (1-го числа, месячные задачи) и `cron/launch-weekly` (каждую среду, задача «Оптимизация рекламы» №10), оба читают регулярные сделки CRM, idempotent, Cloud Scheduler, dry-run. Своё хранилище не требуется. ← блокер для cutover.
7. **Cutover (до 1 июля 2026)**: инвентаризация и **отключение** существующих повторяющихся шаблонов задач Bitrix; dry-run сверка июля; включение Scheduler. См. §8.4.
8. **Финансы в мониторинге**: бюджеты из родителей + суммы сделок/цены товаров → агрегаты на overview.
9. **Интеграция Т‑Банк** (§9): приём платежей (вебхук/опрос выписки), сопоставление по ИНН, автоперевод сделки NEW→EXECUTING, дедуп по operationId.
10. **Деплой**: контейнер на Cloud Run, секреты (вебхук Bitrix + ключ Т‑Банка) из Secret Manager, статика на Firebase Hosting с rewrite `/api/*`.
11. (опц.) **Вебхук-события**: `ONTASKUPDATE` → инвалидация кэша/лента изменений.
12. (опц.) **Аналитический модуль метрик эффективности** (§10): `GET /staff-metrics`, агрегации уровней 1–3 из `tasks.task.list` (`WITH_RESULT_INFO`/`WITH_TIMER_INFO`), справочник нормативов по типам задач, экран «Сотрудники/Команда». Уровни 4–5 — ручной ввод/импорт из журнала гипотез. Независим от планировщика и cutover.

Т‑Банк-интеграция (этап 9) независима от планировщика — её можно делать параллельно и не блокирует cutover 1 июля.

**Срочность:** этапы 1–7 должны быть готовы и проверены **до 1 июля 2026** (≈3 недели от текущей даты), иначе либо пропуск месяца, либо дубли. Если не успеваем — оставить автоматику Bitrix включённой ещё на июль и перенести cutover на 1 августа.
