# Древо услуг и рутинные задачи: SEO и PPC

Бизнес-анализ связки **«услуга из CRM-каталога → набор рутинных подзадач в проекте»** для проектирования архитектуры БП (автогенерация месячного плана работ).

Источники: [crm-products.md](crm-products.md) (каталог услуг), [rutinnye-podzadachi-soderzhimoe.md](rutinnye-podzadachi-soderzhimoe.md) (эталоны подзадач), [akademia-sporta-task-examples.md](akademia-sporta-task-examples.md) (структура месячного цикла), [struktura-opisaniy-zadach.md](struktura-opisaniy-zadach.md).

---

## 1. Главный вывод

Бизнес-модель агентства = **подписка на ежемесячное сопровождение по услуге**. Каждая услуга в CRM (сделка/товар) разворачивается в проекте Bitrix24 в **одну глобальную месячную задачу-родитель** + **типовой набор рутинных подзадач**. Состав и количество подзадач **детерминированы услугой и её пакетом** — значит, генерацию месячного плана можно полностью шаблонизировать.

Ключевая находка: **пункты в описании товара 1:1 соответствуют рутинным подзадачам.** Описание SEO-пакета (товар 87) — это, по сути, чек-лист подзадач месяца:

| Блок в описании услуги (товар 87) | Рутинная подзадача |
|-----------------------------------|--------------------|
| Контентная оптимизация → Сбор семантики | **Сбор семантического ядра** |
| Контентная оптимизация → Подготовка текстов | **Создание ТЗ для написания текста** |
| Контентная оптимизация → Публикация контента на N страницах | **Создание страниц** (×N по пакету) |
| Контентная оптимизация → Внутренняя перелинковка | **Оптимизация перелинковки** |
| Усиление ссылочной массы (off-page) → Размещение ссылок | **Закупка ссылок** |
| Аналитика и отчётность → Подготовка отчёта | **Формирование ежемесячного отчёта (SEO)** |
| Техническая оптимизация и контроль | (входит в сопровождение родителя; точечно — тех. подзадачи) |

То же для PPC: «Настройка и запуск РК», «аналитика», «масштабирование/отключение связок» → подзадачи **Запуск рекламной активности** + **Ежемесячный отчёт** + точечные тех. задачи.

## 3. Древо услуг → рутинные задачи

```
АГЕНТСКИЕ УСЛУГИ (подписка, помесячно)
│
├── 🟦 SEO-ПРОДВИЖЕНИЕ
│   │   CRM: 87 / 1805 / 1801 (пакеты S/M/L), 499
│   │   Родитель месяца: «<Проект> — SEO-продвижение (<Месяц Год>)»
│   │   Параметры родителя: Сайт, Регион, Рекламный бюджет, Доступы
│   │
│   ├── РУТИННЫЕ ПОДЗАДАЧИ (каждый месяц):
│   │   ├── Сбор семантического ядра ............... 1× (под план страниц)         [статична + регион/направление]
│   │   ├── Создание ТЗ для написания текста ....... = N страниц пакета            [предмет ТЗ]
│   │   ├── Создание страниц ....................... N = 3 (S) / 5 (M) / 10 (L)     [N, ссылка на файл услуг]
│   │   ├── Оптимизация перелинковки ............... 1×                            [полностью статична]
│   │   ├── Закупка ссылок ......................... 1× (в рамках бюджета)         [полностью статична]
│   │   └── Формирование ежемесячного отчёта (SEO) . 1× (закрытие месяца)          [Месяц Год]
│   │
│   └── РАЗОВЫЕ / ПО ТРИГГЕРУ (не каждый месяц):
│       ├── SEO-аудит сайта (CRM 1776) ............. при старте проекта
│       ├── Технические правки (микроразметка, каноникал, alt, H-теги, sitemap)
│       └── Тех. оптимизация / правки на сайте (передаётся в Программирование/Админ.)
│
└── 🟥 PPC-ПРОДВИЖЕНИЕ / КОНТЕКСТНАЯ РЕКЛАМА
    │   CRM: 477 / 1809
    │   Родитель месяца: «<Проект> — PPC-сопровождение (<Месяц Год>)»
    │   Параметры родителя: Сайт, Регион, Рекламный бюджет, Доступы,
    │                       UF_CRM_CONV_DATA (Да/Нет/Не выяснено — может ли клиент давать данные о конверсиях)
    │
    ├── РУТИННЫЕ ПОДЗАДАЧИ (каждый месяц):
    │   ├── Запуск рекламной активности ............ 1×+ на площадку/направление    [шапка перем.; чек-листы статичны]
    │   │     ├─ Яндекс Директ (ЕПК: Поиск / РСЯ / Конверсии-трафик)
    │   │     ├─ Google Ads (Поиск / Дисплей)
    │   │     └─ VK (для таргет/догоняющей)
    │   └── Формирование ежемесячного отчёта (PPC) .. 1× (закрытие месяца)          [Месяц Год]
    │
    ├── ЕЖЕНЕДЕЛЬНЫЕ ПОДЗАДАЧИ (под текущего месячного PPC-родителя):
    │   ├── Оптимизация рекламы Я.Директ ........... каждую среду, дедлайн пт +2 дня  [статична; см. шаблон №10]
    │   └── Передача конверсионных сигналов (feedback) каждый раб. понедельник, дедлайн ср +2 дня
    │         (офлайн-конверсии CRM→площадки)        [статична; ТОЛЬКО если UF_CRM_CONV_DATA=Да; см. шаблон №11]
    │
    ├── РЕГУЛЯРНЫЕ, НО ВНЕПЛАНОВЫЕ (по триггеру с созвона, см. шаблон №9):
    │   └── Обновление / запуск РК по запросу .......... частота не фиксирована
    │         ├─ обновить баннеры / креативы
    │         ├─ сменить оффер / цену в объявлениях
    │         ├─ запустить новую акцию (сезонная, Black Friday)
    │         └─ анализ работы РК / оптимизация стоимости заявки
    │
    └── РАЗОВЫЕ / ПО ТРИГГЕРУ:
        ├── Пополнение баланса (CRM 425 / 1813) ..... по необходимости (биллинг)
        ├── Установка антибот-настроек (DNS-экран, Killbit) . при защите от скликивания
        ├── Подключение ROISTAT / счётчиков / Метрики / GTM .. при старте/донастройке
        └── Чистка площадок / обновление РК / анализ работы РК
```

> **Класс «регулярные, но не планируемые».** Задачи обновления/запуска РK по запросу — это рутинный *тип* (повторяется почти каждый месяц), но **не рутина по расписанию**: повод и содержание диктуются динамикой маркетинговых активностей заказчика и фиксируются на еженедельных/периодических созвонах. В БП они НЕ попадают в ежемесячный авто-генератор, а создаются по триггеру «договорённость на созвоне» из готового шаблона (см. шаблон №9 в [rutinnye-podzadachi-soderzhimoe.md](rutinnye-podzadachi-soderzhimoe.md)). Рекомендуемая реализация — кнопка/быстрый шаблон в карточке проекта.

## 4. Правила привязки (для шаблонизатора БП)

При заведении услуги в проекте генерируется месячный план по правилам:

**SEO (пакет → количество):**
- `Создание страниц`: N = {S→3, M→5, L→10} (берётся из выбранного товара-пакета).
- `Создание ТЗ для написания текста`: по числу страниц, требующих копирайта (обычно = N или меньше, если часть страниц технические).
- `Сбор семантического ядра`, `Оптимизация перелинковки`, `Закупка ссылок`, `Ежемесячный отчёт`: ровно по 1 шт/мес независимо от пакета.
- Все тексты подзадач — **константы**, подставляются только: `{{N}}`, `{{ссылка на файл услуг}}`, `{{регион}}`, `{{Месяц Год}}`.

**PPC:**
- `Запуск рекламной активности`: 1 задача на каждое согласованное направление/площадку (Яндекс Директ / Google Ads / VK). Чек-листы в спойлерах — константы; переменна только шапка (Направление, Площадки, Бюджет, Период, Офферы). **Напоминание:** при создании генератор ставит исполнителю напоминание за сутки до дедлайна в 10:00 (`reminderAt = DEADLINE − 1 день, 10:00`; реализация через `calendar.event.add` с `remind` — см. шаблон №7).
- `Передача конверсионных сигналов (feedback)`: **еженедельно** (каждый рабочий понедельник, дедлайн +2 дня/среда), **гейтится полем сделки `UF_CRM_CONV_DATA`** (`crm.deal.userfield.add`, тип `enumeration`: Да/Нет/Не выяснено). `Да` → спавнить; `Нет` → пропустить; `Не выяснено` → вместо неё разовая задача АМ «выяснить возможность передачи данных». Подробности — шаблон №11.
- `Ежемесячный отчёт (PPC)`: 1 шт/мес.
- Мультирегион (товар 1809) → запуск/ведение РК масштабируется на число регионов.

**Общие параметры родителя месяца** (оба направления): `Сайт`, `Регион`, `Рекламный бюджет (Месяц Год)`, `Доступы` — наследуются из карточки проекта/сделки.

## 6. Наблюдения и рекомендации

1. **Высокая стандартизация → готово к автогенерации.** 6 рутинных подзадач SEO и 2 PPC покрывают основную массу месячной работы. Их можно хранить как шаблоны (`tasks.task.add`) с 3–4 плейсхолдерами и спавнить пачкой при создании сделки/наступлении месяца.
2. **Объём SEO привязан к пакету (S/M/L)** — это единственный реальный «переключатель» количества задач. Стоит вынести `N страниц` в свойство товара/сделки и пробрасывать в генератор.
3. **«Закупка ссылок» и «Перелинковка» — полностью статичны** (0 плейсхолдеров) → идеальные кандидаты на полную автоматизацию без участия менеджера.
4. **«Ежемесячный отчёт»** есть в обоих направлениях, текст идентичен (меняется только период) → один общий шаблон с параметром услуги.
5. **Разделение рутины и триггерных задач.** Тех. правки сайта, антибот, ROISTAT, пополнение баланса — не рутина расписания, а реакция на событие; их не надо генерировать автоматически каждый месяц.
6. **Пробел в данных:** у многих товаров `PRICE = 0` и нет привязки `SECTION_ID` к разделу каталога — каталог не структурирован по направлениям. Для чистой архитектуры стоит сгруппировать товары в разделы «SEO / PPC / Разработка / Поддержка» и проставить цены пакетов.
7. **Несогласованность ТЗ-задач:** число «Создание ТЗ» и «Создание страниц» не всегда совпадает (часть страниц технические/без копирайта). В генераторе развести: `страниц с копирайтом` vs `технических страниц`.
8. **PPC ≠ копия SEO по числу задач (реалии 2026).** Пакетные/автоматизированные РК (PMax, ЕПК с автотаргетом, tCPA/tROAS) «съели» ручную середину: управление ставками вручную нерелевантно, а частое вмешательство вредит обучению стратегий. Поэтому PPC-рутина остаётся компактной; единственный высокоценный рычаг человека, который раньше делался нерегулярно, — **передача конверсионных сигналов из CRM в площадки** (см. шаблон №11). Условные по составу услуги задачи (напр. оптимизация Google Ads — есть лишь у ~20% клиентов) генерировать только при наличии соответствующего товара в сделке, а не в общем месячном наборе.

## 8. Связь генерации задач с оплатой счетов (предоплатная модель)

**Биллинг агентства — предоплата:** счёт за месяц M выставляется заранее (в конце месяца M−1), работа по месяцу стартует **после оплаты**. Отсюда главное правило: *работа до оплаты = работа бесплатно*, поэтому **оплата счёта гейтит генерацию месячного пакета**, а не запускается лестница «взыскания после поставки».

### 8.1. Главное правило генератора

> Месячный родитель и его рутинные подзадачи **спавнятся только после того, как счёт за этот месяц получил статус «оплачен»**. До оплаты пакет не создаётся (или родитель висит в статусе «ожидание оплаты» без активных подзадач).

### 8.2. Триггерный флоу месяца

```
[конец месяца M−1]
   Закрыт «Ежемесячный отчёт (M−1)»  ──►  АВТО-ВЫСТАВЛЕНИЕ счёта за месяц M
        │                                  (crm.item.add, смарт-счёт, привязка к сделке;
        │                                   срок оплаты = до 1-го числа месяца M)
        ▼
[T−3 до срока]      ──► авто-напоминание клиенту (+ задача АМ при необходимости)
        ▼
[дата старта M / 1-е число] ── проверка статуса счёта M:
        │
        ├─ ОПЛАЧЕН ──► создать родителя «… (Месяц M)» + авто-спавн всех подзадач
        │              (дедлайны от даты старта; см. разделы 3–4)
        │
        └─ НЕ ОПЛАЧЕН ──► пакет НЕ спавнить; родитель = «ожидание оплаты»;
                          запустить лестницу напоминаний (8.3);
                          уже идущие РК — grace-период (8.4)
        ▼
[оплата пришла позже]  ──► до-генерация пакета с пересчётом дедлайнов
                           от ФАКТИЧЕСКОЙ даты оплаты; money-out задачи разблокируются
```

### 8.3. Лестница напоминаний (до и после срока)

```
T−3 до срока оплаты → авто-напоминание клиенту
Срок оплаты (due)   → проверка статуса; пакет месяца не стартует, пока не оплачено
due +2 дня          → задача АМ «связаться по оплате»
due +5 дней         → эскалация руководителю + флаг на сделке/компании «просрочка»
```

### 8.4. Непрерывность услуги (чтобы предоплата не рвала сервис)

Предоплатный гейт не должен «убивать» уже идущие кампании в первые дни месяца:

- **PPC:** живые РК продолжают откручиваться на остатке баланса — это естественная «мягкая пауза»: без оплаты не выполняется задача **«Пополнение баланса»** (money-out, тип A), и кампания затухает сама по мере исчерпания бюджета, без ручной остановки в эфире.
- **SEO:** не спавним новый месячный объём; ранее опубликованное продолжает работать (накопленный эффект не теряется мгновенно).
- **Жёсткая остановка** (снятие РК с публикации, заморозка проекта) — только **решением человека** после `due +5…+7`, не автоматически.

### 8.5. Что читает генератор (сверено с порталом `leademy.bitrix24.ru`)

**Сущность счёта — смарт-счёт** (`entityTypeId = 31`), единственная воронка: категория `9` «Общая». Классические счета (`crm.invoice.*`) **не используются** (0 записей) — читать только смарт-счета через `crm.item.list` / `crm.item.get`.

**Определение «оплачено» — по стадии.** Стадии воронки (`ENTITY_ID = SMART_INVOICE_STAGE_9`):

| stageId | Семантика | Название |
|---|---|---|
| `DT31_9:N` | — (process) | Новый |
| `DT31_9:S` | — (process) | Отправлен клиенту |
| **`DT31_9:P`** | **`S` (success)** | **Оплачен** ← сигнал оплаты для гейта |
| `DT31_9:D` | `F` (fail) | Не оплачен |

> Гейт проверяет: счёт «оплачен» ⇔ `stageId == "DT31_9:P"` (надёжнее — по `SEMANTICS == "S"`, чтобы не сломаться при переименовании стадий).

**Ключевые поля смарт-счёта:**
- `closedate` — **Срок оплаты** (due date для лестницы напоминаний 8.3).
- `opportunity` — Сумма счёта.
- `parentId2` — **Сделка** (привязка к проекту); `companyId` / `contactId` — компания/контакт; `assignedById` — ответственный.
- ⚠️ **Период (месяц) НЕ закодирован в названии** — `title` это коды клиентов («КМ-19», «ПП-17», «ДС-19»). Сопоставлять счёт с месяцем M нужно по `closedate` / дате создания и привязке `parentId2`, **не парсингом названия**.

**⚠️ Риск данных — дисциплина стадий (блокер для гейта).** На момент сверки из 510 счетов: **422 в «Новый»**, **88 в «Оплачен»**, **0** в «Отправлен» и «Не оплачен». То есть большинство счетов не переводятся дальше «Новый», даже если оплачены. Прямое следствие: гейт «оплачен?» по стадии будет давать **ложное «не оплачено»** для фактически оплаченных счетов, оставленных в «Новый», и ошибочно блокировать работу. Перед включением жёсткого гейта необходимо одно из:
1. **Навести порядок со стадиями** — обязательный перевод оплаченных в «Оплачен» (лучше автоматически: робот/БП по факту поступления платежа двигает стадию).
2. **Детектировать оплату другим сигналом**, если стадии не ведутся дисциплинированно (например, по факту платежа/банковской выписке, привязанной к счёту), а стадию обновлять автоматически.

Без этого предоплатный гейт (раздел 8.1) включать нельзя — он будет тормозить добросовестных клиентов.

**Дашборд.** Те же поля — слой мониторинга: *проект X · счёт (сделка 890) в «Новый», срок оплаты 25.06 · пакет в «ожидании» · РК на остатке баланса*. Заодно дашборд подсветит сам разрыв дисциплины: «422 счёта зависли в Новый» — операционная проблема, видимая владельцу.

**Опционально** — флаг «хронический неплательщик» на компании (`companyId`): для таких ужесточить (без grace), для проверенных — короткий grace по решению АМ.

### 8.6. Классификация задач по финансовой экспозиции (для гейта)

| Тип | Задачи | Поведение при неоплате |
|---|---|---|
| **A. Агентство тратит деньги клиента** | Закупка ссылок, Пополнение баланса (425/1813) | **Жёсткий гейт всегда** — не выполнять без подтверждённой оплаты |
| **B. Труд агентства** | Семантика, ТЗ, страницы, перелинковка, запуск/оптимизация РК, feedback-сигналы | В предоплате — **не спавнить до оплаты**; уже идущее не глушим резко (grace, 8.4) |
| **C. Отчётность** | Ежемесячный отчёт SEO/PPC | Не гейтить; закрытие отчёта M−1 = **триггер счёта за M** |

> **Не автоматизировать:** автоснятие живых РК с публикации по факту неоплаты (операционный/репутационный риск) — только задача-решение человеку. Жёсткий гейт всего пакета оправдан именно в предоплатной модели, но непрерывность по 8.4 обязательна, иначе теряем клиента из-за одного задержанного платежа.

### 8.7. Автоматический детект оплаты через T-Business API

Решает корневую проблему из 8.5 (422 счёта зависли в «Новый»): стадию «Оплачен» проставляет **робот по факту поступления на расчётный счёт**, а не человек. Только тогда предоплатный гейт (8.1) применим.

**Важно — какой именно API.** Нужен **T-Business API (T-API), раздел «Счета и выписки»** — поступления на расчётный счёт. НЕ интернет-эквайринг (тот про приём карт на витрине, у нас оплата по счёту).

**Модель доступа.** T-API использует не классические OAuth-scopes, а **«доступы» (access templates)** — строки `opensme/inn/[{inn}]/kpp/[{kpp}]/<ресурс>/<действие>`. Токен выпускается в **ЛК Т-Бизнес → Интеграции → Выпуск токена** (галочки доступов). Проверка выданного — `POST https://id.tbank.ru/auth/introspect` (поле `scope`). Срок жизни токена H2H — **90 дней с последнего использования**.

**Доступы для токена (минимально необходимые, только чтение):**

| Доступ в UI («Счета и выписки») | Access template |
|---|---|
| Информация об операциях компании | `opensme/inn/[{inn}]/kpp/[{kpp}]/bank-statements/get` |
| Информация о счетах компании | `opensme/inn/[{inn}]/kpp/[{kpp}]/bank-accounts/get` |

> ⚠️ **Минимальные права:** НЕ выдавать токену доступы на создание/проведение платежей (списание) — для детекта поступлений не нужны, лишний риск. Метод `…/account-operations/get` устаревший (3 rps) — использовать `bank-statements/get`.

**Способ доставки:**
- **Polling выписки (базовый, рекомендуется)** — периодический опрос `GET /api/v1/statement` (раз в 15–30 мин достаточно для гейта). Общедоступно, без согласований. Лимит ~10 rps.
- **Webhook о новых операциях (push, опционально)** — банк шлёт JSON на публичный HTTPS-эндпоинт. Пока **пилот, только для прямых H2H**, подключается **по письму на `openapi@tbank.ru`** (ИНН + URL). Доставка с подсети `91.194.226.0/23`, ретраи exponential backoff до 24 ч, дубли возможны (но `operationId` уникален). Приём webhook токена не требует, но обогащение — через ту же выписку.

**Контур (платёж → счёт → стадия):**
```
[T-API: новая входящая операция / выписка]
   сумма, дата, ИНН плательщика, назначение платежа, operationId
        │
        ▼
[Матчинг с смарт-счётом Bitrix (entityTypeId=31, стадия ≠ Оплачен)]
   приоритет 1: номер счёта в «назначении платежа»
   приоритет 2: ИНН плательщика (компания) + сумма (opportunity) + окно дат (closedate ± N)
        │
   ├─ однозначный матч ──► crm.item.update: stageId → DT31_9:P («Оплачен»)
   │                        + (опц.) комментарий с operationId для идемпотентности
   │                        ──► разблокировка гейта 8.1 для месяца проекта
   └─ неоднозначно/нет матча ──► задача бухгалтеру/АМ «разнести платёж вручную»
```

**Условия надёжного матчинга:**
- Просить клиентов указывать **номер счёта в назначении платежа** (приоритетный ключ) — иначе матч по ИНН+сумме+дате, который ломается при равных суммах у разных счетов одной компании.
- **Идемпотентность:** хранить обработанные `operationId` (дубли webhook/повторный polling не должны двигать стадию повторно).
- Частичная оплата (сумма операции ≠ `opportunity`) — не закрывать счёт автоматически, отправлять на ручную разноску.

> Итог: токен T-API выпускается с двумя read-доступами раздела «Счета и выписки» (`bank-statements/get`, `bank-accounts/get`); робот матчит поступления со смарт-счетами и двигает их в `DT31_9:P`. Это и закрывает дисциплинарный разрыв 8.5, и делает предоплатный гейт 8.1 рабочим.

Источники: [Счета и выписки](https://developer.tbank.ru/docs/api/scheta-i-vipiski) · [Получить выписку по счету](https://developer.tbank.ru/docs/api/get-api-v-1-statement) · [Авторизация и токены (FAQ)](https://developer.tbank.ru/docs/support/faq/authorize) · [T-API для бизнеса](https://www.tbank.ru/business/open-api/)

> **Стадия сделки = «в работе».** Детект оплаты двигает не только смарт-счёт в `DT31_9:P`, но и **сделку** воронки «Производство» (category 0) из `NEW` «Получение оплаты от клиента» → `EXECUTING` «В работе». Стадия `EXECUTING` — машинный признак «проект оплачен и идёт в работу» для гейта и дашборда.

### 8.8. Клиенты с наличной оплатой (обход банковского детекта)

**Проблема:** оплата наличными **не появляется в выписке T-API** (8.7), поэтому робот никогда не переведёт такого клиента в «Оплачен» / «В работе» — и предоплатный гейт его навсегда заблокирует.

**Решение — флаг клиента.** На карточке **компании** заводится признак «оплата наличными»; для помеченных клиентов гейт **не ждёт банковский детект**, а автоматически переводит сделку `NEW → EXECUTING` («В работе») и смарт-счёт → `DT31_9:P` при наступлении периода / выставлении счёта.

| Параметр | Значение |
|---|---|
| Где | поле **компании** (`crm.company.userfield.add`, scope `crm`) |
| Предлагаемый код | `CASH_PAYER` → `UF_CRM_CASH_PAYER` |
| Тип | `boolean` (да/нет). Дефолт «нет» здесь безопасен: нет флага = обычный путь через T-API-детект, а не тихая блокировка |

**Логика генератора:**
- `UF_CRM_CASH_PAYER = Y` → пропустить ожидание T-API; авто-перевод сделки в `EXECUTING` и счёта в «Оплачен» в момент старта периода.
- `N` / пусто → стандартный детект оплаты по выписке (8.7).

> **Создано в CRM** портала `leademy`: поле компании `UF_CRM_CASH_PAYER` (boolean, userfield ID `1409`).

> ⚠️ **Риск (честно):** наличные ещё менее проверяемы, чем банковский перевод — авто-перевод «в работе» по флагу означает работу до подтверждённого факта получения денег. Более безопасный вариант флага: он не «всегда в работе», а **«не ждём банк; перевод в работу по ручному подтверждению получения налички за период»** (галочка/действие бухгалтера раз в период). Рекомендую этот вариант; если нужен именно полностью автоматический перевод — оставить как выше, но осознавая дебиторский риск.

## 9. Актуализация списка проектов (жизненный цикл)

Список активных проектов не статичен: клиенты приходят и отваливаются. Дашборду и генератору нужен **самоподдерживающийся актуальный список**, иначе задачи будут спавниться на ушедших клиентов или не создаваться для новых.

**Источник истины — сделки, не группы.** Рабочие группы теперь читаются (`sonet_group.get` доступен, scope `sonet_group` выдан) — это реестр проектов и место, где живут задачи. Но **признак активности проекта брать из них нельзя**: на момент сверки **54 группы, все `ACTIVE=Y`, `CLOSED=0`** — флаг закрытия не ведётся, ушедшие клиенты остаются «активными» группами (та же дисциплинарная дыра, что со счетами в «Новый»). Поэтому:
- **Активность проекта определяется по сделкам** воронки «Производство» (category 0) и продлениям (category 7) — см. 9.1–9.2.
- **Группа = контейнер задач + связь проекта**; её `CLOSED` следует проставлять **роботом при отваливании** (9.2), а не полагаться на ручное закрытие. Иначе список «активных» групп всегда завышен.

### 9.1. Проект ПОЯВЛЯЕТСЯ
- **Триггер:** сделка попадает в воронку «Производство» (создана в ней / выиграна в продающей воронке и породила производственную) и/или доходит до `EXECUTING` «В работе».
- **Действие генератора:** завести проект — (при наличии прав) рабочую группу + месячного родителя + рутинные подзадачи по услуге/пакету (разделы 3–4). Для предоплаты — после оплаты (раздел 8) либо сразу для cash-клиентов (8.8).

### 9.2. Проект ОТВАЛИВАЕТСЯ
- **Триггеры:**
  - сделка → `LOSE` «Сделка провалена»;
  - **продление не оплачено** — нет новой оплаченной сделки/счёта на следующий период (воронка «Продажа продления», category 7) в течение N дней после `closedate`;
  - явный признак расторжения (стадия/флаг).
- **Действие:** пометить проект неактивным → **прекратить авто-генерацию** месячных/недельных задач, снять напоминания, остановить money-out задачи; перевести в архив (не удалять — историю и отчёты сохранить).

### 9.3. Механизм синхронизации
- **Событийно (реактивно):** подписка на события сделок — `ONCRMDEALADD`, `ONCRMDEALUPDATE` (смена стадии) — мгновенно отражает появление/отваливание.
- **Сверка (проактивно), раз в сутки:** полный проход сделок «Производство» + сопоставление со списком активных проектов генератора → закрыть расхождения (дрейф событий, ручные правки). Это же — индикатор на дашборде: «новых проектов за неделю: N», «отвалилось: M».
- **Идемпотентность:** повторное событие/сверка не должны плодить дубли родителей и подзадач (ключ — сделка↔проект↔период).

> Связь с биллингом: «отваливание по неоплате продления» (9.2) — это та же предоплатная логика (раздел 8), только на горизонте месяца: нет оплаченного счёта на следующий период → проект уходит из активных, а не просто «пакет в ожидании».

### 9.4. Готовность данных к автоматизации — сверка на 2026-06-19 (БЛОКЕРЫ)

Фактическая проверка портала показала: **ни один статус-сигнал сейчас не ведётся дисциплинированно**, поэтому автогейт (8) и автосписок проектов (9) во включённом виде работать не будут. Это надо чинить ДО запуска автоматики.

| Сущность | Ожидаемый сигнал | Факт | Вывод |
|---|---|---|---|
| Рабочие группы (54) | `CLOSED=Y` = отвалился | 54 активных, `CLOSED=0` | по группам нельзя ловить отток |
| Сделки «Производство» (352) | `EXECUTING` = в работе | 296 `WON`, 1 `EXECUTING`, 19 `NEW`, 24 `LOSE` | стадия «В работе» фактически не используется |
| Смарт-счета (510) | стадия «Оплачен» | последний `DT31_9:P` — **25.01.2026** (≈5 мес назад); 422 в «Новый» | ручная отметка оплаты заброшена |
| Связь группа↔компания | общий ключ | по имени матчится **8 из 54** | структурной связи проект↔клиент нет |

**Что это значит и что чинить раньше автоматики:**

1. **Нужен структурный ключ связи `проект (группа) ↔ компания/сделка`.** Сейчас его нет (имя не работает: бренды групп ≠ юрлица). Решение — UF-поле на группе со ссылкой на компанию/сделку (или хранить `groupId` на сделке). Без этого «проект → клиент → счёт → платёж» не соединить.
2. **T-API-робот оживляет мёртвый сигнал оплаты.** Хорошая новость: автоматический детект из выписки (8.7) — это и есть замена заброшенной ручной дисциплины «Оплачен». После внедрения стадия счёта снова станет достоверной (её будет двигать робот, а не человек).
3. **«Активность проекта» определять поведением, а не статусом** — по факту свежей оплаты за период (после оживления сигнала по п.2), а не по `EXECUTING`/`CLOSED`, которые не ведутся.

> Итог сверки: текущая автоматизация (разделы 8–9) **не может быть включена на сегодняшних данных**. Порядок внедрения см. ниже — **порядок приоритетов**.

**Порядок реализации (приоритеты, решение владельца 2026-06-19):**
1. ✅ **Структурная связь группа↔компания** (`UF_CRM_GROUP_ID`) — сделано, см. §9.5.
2. **Мониторинг-слой на дашборде** — видимость дрейфа (неархивные группы, счета в «Новый», 5 мес без «Оплачен»), список активных проектов по структурной связи + ручным сигналам. Не требует T-API.
3. **Генератор рутинных задач** (разделы 3–7) — автоспавн месячных/недельных подзадач по услуге/пакету; гейт оплаты — в **мягком/ручном режиме** (стадию «Оплачен»/«В работе» двигают вручную или по `UF_CRM_CASH_PAYER`).
4. **(ПОСЛЕДНИЙ ПРИОРИТЕТ) T-API токен + робот оплаты** — оживляет сигнал «Оплачен» по банковской выписке (доступы `bank-statements/get` + `bank-accounts/get`), после чего включается **жёсткий автогейт** (раздел 8) и автосписок проектов на достоверных данных.

> **Следствие отсрочки T-API:** пока шаг 4 не сделан, достоверного автосигнала оплаты нет → предоплатный гейт (8.1) и «активность по факту оплаты» (9) работают в ручном/полуавтоматическом режиме. Это осознанный компромисс: сначала генератор и мониторинг (ценность без T-API), банковский автодетект — в конце.

### 9.5. Реализация структурной связи `проект ↔ клиент`

**Готовых полей связи нет** (проверено): у рабочей группы (`sonet_group`) нет CRM-привязки; у сделки — десятки UF-полей интеграций, но поля «группа/проект» нет. Однако **связь уже частично живёт в данных**:
- **Мост через задачи:** рутинные задачи несут `UF_CRM_TASK = ["D_<dealId>"]`. Проверено сквозь: группа 29 → задача `D_129` → сделка 129 → `COMPANY_ID 93` → 37 ежемесячных счетов (74 000 ₽). Цепочка замыкается.
- **Покрытие авто-связи:** из 54 групп **26 связываются автоматически** из существующих задач (есть `D_` в задачах), **28 — нет** (старые задачи без привязки).

**Кардинальность: 1 проект (группа) : N компаний.** У одного клиента может быть несколько **юридических лиц** — он платит через разные компании (разные ИНН). Значит:
- **Якорь проекта — группа** (единая стабильная сущность проекта).
- **Стабильный коммерческий хаб — компания** (сделки и счета несут `companyId`; сделки транзитны и продлеваются через `UF_CRM_PREVIOUS_DEAL`, компания живёт дольше). Но компаний у проекта может быть несколько.
- **Активность/оплата проекта = агрегат по ВСЕМ юрлицам группы.** Клиент может в один период платить через юрлицо A, в другой — через B. Матчинг: `платёж → компания → группа`; «оплачено за период» = заплатило *любое* из юрлиц проекта. Период счёта — по `closedate` (у абонемента помесячно).

**Что добавить — ключ `компания → группа` (N:1):**
- **Рекомендуется** — UF на **компании** `UF_CRM_GROUP_ID` (integer = ID рабочей группы), через `crm.company.userfield.add` (метод уже проверен). Каждое юрлицо указывает на свой проект; несколько юрлиц → одна группа. Резолв в обе стороны: `компания → группа` напрямую, `группа → все юрлица` = фильтр компаний по `UF_CRM_GROUP_ID`.
- Поле на группе (`SONET_GROUP`) одиночным значением **не подходит** — не вместит несколько юрлиц (только если множественное поле-список, что сложнее).
- ⚠️ Крайний случай M:N (одно юрлицо обслуживает несколько проектов) данными пока не подтверждён; если появится — `UF_CRM_GROUP_ID` сделать множественным или вынести связь в отдельную таблицу-маппинг.

**Бутстрап (минимум ручного):**
1. **Авто:** по каждой группе собрать `UF_CRM_TASK` задач → `crm.deal.get` → `COMPANY_ID` → проставить `UF_CRM_GROUP_ID = <ID группы>` этой компании. Закрывает ~26/54 сразу (по основному юрлицу).
2. **Добор юрлиц:** у проекта может быть несколько компаний — остальные юрлица того же клиента доцепить по группировке (общий ИНН-холдинг / контакт / название) с подтверждением человеком.
3. **Ручной (one-time):** оставшиеся ~28 групп без авто-связи — матч по названию как список-подсказка на подтверждение.
4. **На будущее:** генератор при создании проекта **всегда** проставляет `UF_CRM_TASK = D_<dealId>` на каждую рутинную задачу (чинит текущую неполноту) и `UF_CRM_GROUP_ID` на компанию(-и) проекта.

**Что это открывает:** полный джойн `группа → {юрлица-компании} → {сделки, счета}` → работают гейт (8) и автосписок (9); «активность проекта» = за период оплатил счёт **любой** из его юрлиц (после оживления сигнала T-API-роботом, п.2 из 9.4); продления отслеживаются по `UF_CRM_PREVIOUS_DEAL`.
